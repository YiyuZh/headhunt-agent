## 关键决策

- 项目主目录以当前 clone 的 `<repo-root>` 为准，不再使用 Downloads 作为项目工作目录；当前 Windows clone 为 `D:\apps\headhunt-agent`，历史 macOS/Linux clone 为 `/Users/w/Documents/lietou`。
- 文档采用规范目录结构：根目录放项目记忆和索引，PRD 放 `docs/prd/`，模块化 PRD 放 `docs/prd/modules/`。
- 完整 PRD 由模块化 PRD 拼接生成，并同步为：
  - `docs/prd/AI猎头作战系统_PRD_完整拼接版.md`
  - `docs/prd/ai猎头_agent体系_codex搭建说明.md`
- 产品核心不是多 Agent 展示，而是 AI 猎头作战系统：飞书任务输入、结构化任务确认、需求会审、人才地图、候选人证据筛选、触达/报告、跟进复盘。
- 运行核心采用 `FastAPI + LangGraph + policy-engine + AgentHarness + ArtifactStore + Gateway + 飞书 War Room + ReviewGate + AgentSOPRegistry`。
- Agent 不能直接互相调用，必须通过 ArtifactStore 交换结构化产物。
- Agent 不能直接访问公网、数据库、飞书、Discord optional adapter、模型 Key，必须通过授权 Gateway / Registry。
- Agent 不能自由 agent-to-agent 聊天；三省六部是同一 Canonical Context 下的角色审查矩阵。
- 用户输入先由 `TaskIntakeParser` 解析成 `CanonicalTaskBrief / RequisitionBrief / CandidateEvidencePack` 等结构化 JSON，必须经用户 double check approve 后才能进入正式 graph。
- 下游 Agent 默认不得重新解析原始任务或原始简历，只能消费允许字段、ArtifactRef、MemoryRef、SOPRef 和 SourceRef。
- 长期记忆必须审批后写入，防止越用越偏。
- 统一运行时口径为 `headhunter_war_room_graph -> council_deliberation_graph -> business subgraph`，会审 Agent 不直接铺在根图上。
- 会审层必须输出 `council_mode` 和 `mode_reason`；系统自动判断 `triage | lite | standard | full_council`，用户明确要求“三省六部”时强制 `full_council`。
- 任务授权后的飞书 War Room 进度卡、追问卡、确认卡和结果卡可自动发送；业务表写入、Bitable 同步、报告发布、外部触达、推荐结论和长期记忆 active 仍必须人工确认。
- LangGraph state 只保存 artifact 摘要和引用，完整 artifact 通过 ArtifactStore 的 `content_ref` 读取。
- 第一版主链路直接接入 PostgreSQL、PostgreSQL checkpointer、pgvector、真实飞书事件、飞书交互卡片、飞书 War Room 群消息和 Bitable 同步；mock / fake gateway 仅用于测试和失败隔离。
- 飞书事件入口 `/feishu/events` 必须只做签名校验、challenge、allowlist、幂等落库和 3 秒内 ACK；完整 AI 工作流通过异步 worker 进入 TaskIntakeParser、double check、`headhunter_war_room_graph` 或 resume。
- 飞书 approve/edit/reject 必须走 `/feishu/card-actions` 的卡片回调，解析 `thread_id/action_id/interrupt_id/idempotency_key` 后生成 `HumanApproval` 并 resume；`/human-approval/{thread_id}` 仅作为内部调试或后台 resume API。
- 飞书接入必须有 `feishu_event_logs` + `feishu_card_actions` + `feishu_outbox` + Bitable chunk/record_id 映射的幂等审计，ACK/toast 前先提交 PostgreSQL；使用 `event_id`、`message_id`、`action_id` 和 `idempotency_key` 的数据库唯一约束与原子 insert / claim 防止重复 graph run、重复 resume 和重复副作用。
- Discord 降级为 optional adapter；现有 `DiscordInteractionVerifier`、`/model` BYOK 和 command register 可保留用于历史测试和后续接入，但不再是第一版验收主路径。
- 第一版主链路直接启用 PostgreSQL + pgvector 向量记忆；Qdrant / Milvus 只保留适配接口，不作为首版依赖。
- 复杂记忆系统一步到位：RunMemory 自动写入并向量化；ProjectMemory / AgentMemory / CaseMemory / UserCorrectionMemory / ProceduralMemory 必须审批后 status=active 才进入可检索长期记忆池。
- 记忆时间治理：RunMemory 默认 30 天；长期记忆默认 90 天；重要记忆可人工续期或升级 permanent；过期、撤销、pending_review 不得检索。
- Agent 不能每轮 loop 注入完整历史、完整 `RecruitmentState`、完整 `node_history`、全量 AgentRuns、全量 artifacts 或全量长期记忆；只能通过 `AgentHarness.build_context_pack` 注入当前节点需要的 artifact refs、MemoryRef、SOPRef、source refs 和预算信息。
- ReviewGate 必须在关键产物后运行，输出 `pass | needs_fix | needs_human`；`needs_fix` 最多自动重试一次，第二次失败必须人工确认。
- ReviewGate 是 artifact-level quality gate，不是全局聊天式审查；每个关键 artifact 后必须通过 conditional edge 路由 `pass / needs_fix / needs_human`。
- `needs_fix` 只能回到该 artifact 对应的 repair_node 一次，不得重跑完整 graph 或无关上游节点；第二次仍失败必须进入 `interrupt_human_approval`。
- `TaskIntakeParser` 输出的每个关键字段必须有 `field_source` 和 `confidence`；无 source 的推断只能进入 `assumptions`，不能进入事实字段。
- Double Check approve 后必须冻结 `CanonicalTaskBrief.version`；下游 Agent 只能读取冻结版 `CanonicalTaskBrief`、ArtifactRef、MemoryRef、SOPRef 和 SourceRef。
- AgentSOPRegistry 吸收 Agent SOP、BMAD、Claude Code subagents、Cursor Rules 的方法结构，但不作为运行时外部依赖；每个节点最多注入 1 个主 SOP + 2 个审查 SOPRef。
- `MemoryGateway.retrieve` 必须执行 policy scope filter、tenant/guild/user/project/requisition/candidate scope filter、pgvector 召回、加权/rerank、MMR 去重、token budget 压缩和检索审计。
- 公司背调、当前在职状态、融资、新闻、裁员和组织变化等当前事实必须来自 SearchGateway source_refs，长期记忆不得替代实时搜索。
- 简历关键词抽取以当前简历/JD artifact 为主；长期记忆只提供 taxonomy、项目规则、用户修正和 SOP。
- Phase 0 审查后补充硬约束：飞书消息事件必须使用稳定 `message_id` / `dedupe_key` 防重复启动 graph；Bitable 写入 chunk 必须持久化并传飞书官方 `client_token`；回调必须通过 `FeishuCallbackVerifier` 处理 raw body 签名、timestamp/nonce/signature、解密和 token 校验。
- Phase 0 审查后补充硬约束：`MemoryItem.status` 统一为 `draft | pending_review | active | revoked | expired`；长期记忆只有审批通过后才可 active；全文读取必须写入 `content_access_audits`。
- Phase 0 审查后补充硬约束：`full_council` 必须运行六部并行 + `ChallengeReviewAgent + CouncilSynthesizerAgent`；`ContextPackBuilder` 只能接 typed selector / allowed snapshot，不能接 raw `RecruitmentState`。
- Phase 1 代码骨架历史约束：在 verifier、event log、durable outbox 和幂等 resume 未实现前，回调入口必须拒绝真实事件，避免假 ACK 吞事件；该保护已由 Phase 3 的 verifier + 持久化 outbox 入口替代。
- Phase 1 代码骨架硬约束：`ContextPack` 的 `artifact_refs` 和 `budget_remaining` 使用强类型模型并 `extra=forbid`，防止在嵌套字段塞 raw state/history。
- Phase 2 数据库硬约束：初始 Alembic 迁移必须冻结为显式 `op.create_table` / `op.create_index` / `op.drop_*`，不能导入运行时 ORM `Base.metadata.create_all()` 或 `drop_all()`。
- Phase 2 outbox 硬约束：`claim_next()` 必须同时领取到期 `pending` 和过期 `claimed` lease，避免 worker 崩溃后 outbox 永久卡死。
- Phase 3 飞书回调硬约束：事件订阅回调和新版 `card.action.trigger` 卡片回调都必须使用 Verification Token、Encrypt Key SHA-256 签名和可选 `encrypt` 解密路径；旧版 SHA-1 卡片签名不得作为生产入口。
- Phase 3 outbox dispatcher 硬约束：`claim_next()` 已增加 attempt_count，dispatcher 失败路径不得再次 `+1`；claim 和最终状态必须显式 commit，异常必须 rollback。
- Phase 4 飞书网关硬约束：发送卡片使用 `POST /open-apis/im/v1/messages?receive_id_type=chat_id`、`msg_type=interactive`、`content` JSON string 和 `uuid` 去重；更新已发送卡片使用 `PATCH /open-apis/im/v1/messages/:message_id` 且 body 只发送 `content`。
- Phase 4 Bitable 硬约束：`batch_create` 的 `client_token` 必须是 UUIDv4 并通过 query 传给飞书；不得换 token 盲目重试。
- Phase 5 LangGraph 硬约束：子图作为根图节点时只能显式返回 delta，不得把完整 reducer state 原样返回给父图，避免 `department_opinions`、`artifacts` 和 `node_history` 重复膨胀。
- Phase 5 runtime 硬约束：`graph_dispatch` / `resume` 默认必须拒绝未接真实 AgentHarness、AgentRuns、War Room 卡片和 interrupt/action gate 的生产成功，不能因为 minimal graph 可 invoke 就标记 outbox 成功。
- Phase 5 ArtifactStore 硬约束：`read_content()` 必须先查 artifact 元数据，同时校验 `can_read_artifact_content`、`allowed_artifact_types_read` 和 `pii_access_level`；拒绝读取也要写 denied audit。
- Phase 5 Bitable 硬约束：chunk 成功落库必须强校验 `records / record_ids / entity_refs` 长度和顺序，同一 `client_token` 遇到不同 `payload_hash` 或不同 `record_ids` 必须标记 conflict，不得覆盖成功记录。
- Phase 5 Memory 硬约束：MemoryGateway 只能返回 `MemoryRef`，`owner_agent` 默认只限制 `AgentMemory`；`RunMemory / ProjectMemory / CaseMemory` 主要由 scope + policy 控制；`memory_embeddings.content_hash` 不设唯一约束。
- Phase 6 AgentHarness 硬约束：`AgentHarness.run_agent` 只能接收 typed `AgentTask`，LLMGateway 只能看到 `ContextPack` 和 schema；不得把 raw state、messages、node_history 或完整 AgentRuns 传给模型。
- Phase 6 War Room 硬约束：War Room 卡片只能展示 council_mode、mode_reason、summary、artifact_refs、memory_refs 和 token 估算，不展示 raw prompt 或未授权全文。
- Phase 6 ActionGate 硬约束：ActionGate 只能提出 `ActionProposal` 和确认卡，不得执行业务副作用；重复 `idempotency_key` 必须复用已有 proposal，不得生成新的 action_id/interrupt_id。
- Phase 7 interrupt 硬约束：`interrupt()` 必须放在业务子图之后、业务副作用执行之前；interrupt 节点恢复会从节点开头重跑，所以 ActionGate 必须按稳定 `idempotency_key` 复用已有 proposal，不能重复发确认卡或生成幽灵 action。
- Phase 7 HumanApproval 硬约束：飞书卡片回调必须携带并校验 `thread_id/action_id/interrupt_id/idempotency_key`，且必须匹配 pending `ActionProposal`；同一 action 的 approve/reject/edit 共用同一 `idempotency_key`，避免多终态。
- Phase 7 Bitable 副作用硬约束：`bitable_write` outbox 执行前必须带 `action_id` 并校验 `ActionProposal.status in ('approved','executed')`；reject 不得写业务表。
- Phase 8 worker 硬约束：生产 worker 必须通过真实 dependency factory 装配 AgentHarness、ActionGate、ActionExecutor、FeishuGateway 和 BitableGateway；不得用测试 scaffold graph 或未授权 handler 标记 outbox 成功。
- Phase 9 readiness 硬约束：配置检查只输出环境变量名、配置分组、用途和下一步，不得输出 OpenAI Key、飞书 App Secret、Verification Token 或 Encrypt Key；配置齐全不等于真实权限/网络联调已通过，必须继续写“未验证”。
- Phase 10 PostgreSQL smoke 硬约束：smoke check 可执行 `PostgresSaver.setup()` 初始化/验证 checkpointer 基础设施表，但不得写业务数据；fake engine 单测不能被写成真实 PostgreSQL 联调已通过。
- Phase 11 external smoke 硬约束：Feishu smoke 只能获取 tenant token、读默认群信息、读 Bitable records/search；不得发送 War Room 卡片、不得写 Bitable 记录、不得创建飞书任务或触达候选人。OpenAI smoke 会消耗少量测试 token，必须可用 `--skip-openai` / `--skip-llm` 跳过。
- Phase 12/15 task authorization 硬约束：`/tasks/authorize` 是受保护内部调试入口，默认 `approved=false`，只有显式 `approved=true` 才能表示人工授权并排队 `graph_dispatch`；不得同步运行 graph、不得写业务数据、不得发布报告、不得创建外部任务、不得外部触达、不得提交推荐结论；未传 `source_ref` 时必须生成稳定 `request:<hash>` 和稳定 `thread_id/task_id`，避免重复授权产生重复 outbox 或孤儿 thread；跨任务复用 `thread_id` 必须返回 409。
- Phase 12 outbox 幂等硬约束：`FeishuOutboxWriteRepository.enqueue_json()` 遇到已存在的 `idempotency_key` 时必须返回已有 `payload_ref`；如果同一幂等键对应 payload hash 不一致，必须抛出冲突错误，不能让 API 响应描述新 payload 而 outbox 实际指向旧 payload。
- Phase 13 inspection 硬约束：只读审计查询接口只能返回状态、摘要、refs、pending interrupts 和 token 估算；不得读取 ArtifactBlob 全文、不得返回原始 prompt、不得展开完整 ContextPack。
- Phase 14 approval 硬约束：内部审批接口只能读取 ActionProposal 摘要，并在提交后写 `HumanApproval` 与 `feishu_outbox(kind='resume')`；不得直接调用 ActionExecutor、不得同步执行业务写入；`edit` 必须带非空 `edited_payload`，重复 edit 必须比较 payload，不同编辑内容必须返回冲突；飞书卡片 payload、direct resume payload、graph normalize 和 ActionExecutor 都必须拒绝无编辑内容的 edit。
