## 简历必须突出的能力

### 1. HR 真实业务场景理解

这份 JD 明确写了 HR 真实业务场景。你的项目原始场景就是招聘/猎头，天然比客服版本更贴近。

要突出：

- 岗位需求解析。
- 候选人信息处理。
- 简历/JD 结构化。
- 人才地图和候选人辅助判断。
- 招聘流程效率。
- HR 用户体验。

推荐表达：

> 围绕招聘/HR 场景设计 AI 智能工作流原型，覆盖岗位需求解析、候选人信息处理、结构化确认、AI 辅助判断、人工确认和复盘记忆。

### 2. Python 与自动化工具能力

JD 要的是“通过 Python 及自动化工具构建智能工作流”。所以不能只写 PRD，要写你能把流程自动化。

可突出：

- Python。
- FastAPI。
- LangGraph。
- 自动化脚本。
- outbox worker。
- Docker Compose。
- 数据处理与结构化。

推荐表达：

> 使用 Python、FastAPI、LangGraph 和 PostgreSQL/pgvector 构建自动化工作流，将重复性信息整理、结构化确认、结果生成和记录沉淀流程化。

### 3. AI Agent 工作流能力

这份 JD 直接写了“理解 AI Agent 的基本逻辑”。你项目里最强的匹配点就是 Agent 治理。

要突出：

- 任务拆解。
- 上下文控制。
- 工具调用边界。
- ReviewGate 质量检查。
- HumanApproval 人工确认。
- ArtifactRef / MemoryRef 结构化交接。
- 长期记忆与检索式注入。

推荐表达：

> 理解 AI Agent 的任务拆解、上下文管理、工具调用和人工确认逻辑，设计结构化任务、最小上下文、质量审查和记忆检索机制，提升 AI 工作流稳定性。

### 4. 技术方案落地能力

JD 要你成为“技术与业务的连接者”。简历里要体现你能把业务需求变成文档、接口、表结构和测试。

可突出：

- PRD。
- 产品功能文档。
- 接口文档。
- 数据库设计。
- 测试验收计划。
- 部署手册。
- README。

推荐表达：

> 输出 PRD、接口文档、数据库设计、部署手册和操作手册，将业务需求转化为可开发、可测试、可部署的 AI 产品方案。

### 5. AI 工具熟练度

JD 点名了工具。简历里可以写，但要按真实使用情况，不要假装全部深度掌握。

建议写法：

> 重度使用 ChatGPT / Claude / Codex 辅助需求拆解、代码生成、文档整理和方案复盘；了解 Cursor、Windsurf、Copilot 等 AI 编程工具在脚本生成、代码补全和快速原型中的使用方式；理解 AI Agent 在任务规划、上下文管理和工具调用中的基本逻辑。

如果你确实使用过 Cursor、Windsurf、Copilot、Midjourney，再把它们写进技能栏；如果只是了解，面试时说“了解/试用过”，不要说“精通”。

## 面试讲法

### 如果问：你的项目怎么匹配京东 HR 真实业务场景？

可以答：

> 我的项目原始场景就是招聘/猎头，所以和 HR 业务天然接近。我没有把它包装成真实京东项目，而是把它作为一个可迁移到 HR 场景的 AI 工作流原型。它覆盖岗位需求解析、候选人信息处理、AI 辅助判断、人工确认和复盘记忆，底层逻辑和 HR 场景中的简历处理、招聘流程提效、候选人辅助筛选很接近。

### 如果问：你怎么用 Python 和自动化工具提升效率？

可以答：

> 我会先识别流程里的重复性工作，比如信息整理、字段结构化、结果生成、记录沉淀和状态流转，再用 Python 和工作流框架把它们流程化。我的项目里用 FastAPI 承载接口，用 LangGraph 编排任务，用 PostgreSQL/pgvector 做数据和记忆底座，用 Docker Compose 做部署，把原来靠人工反复整理的信息处理流程变成可追踪的智能工作流。

### 如果问：你理解的 AI Agent 基本逻辑是什么？

可以答：

> 我理解 AI Agent 不只是“模型回答问题”，而是任务拆解、上下文控制、工具调用、结果审查和人工确认的组合。我的项目里没有让 Agent 自由聊天，而是用结构化任务、ArtifactRef、MemoryRef 控制上下文，用 ReviewGate 做质量检查，用 HumanApproval 控制高风险结果，这样更适合真实业务场景落地。

### 如果问：你如何把 HR 需求转成 AI 解决方案？

可以答：

> 我会先把业务需求拆成目标、输入、输出、异常分支和验收标准。例如“提升招聘筛选效率”，不能只写一句做 AI 筛选，而要拆成岗位需求解析、简历字段结构化、候选人匹配依据、AI 结论解释、人工确认和数据复盘。然后再落到 PRD、接口、数据库、工作流和测试验收里。我的项目就是按这个方式沉淀了 PRD、接口文档、数据库设计和部署手册。

### 如果问：你怎么体现逻辑思维和好奇心？

可以答：

> 这个项目过程中我不断追问一些实现问题：Agent 要不要自由互聊、每轮要不要塞全量历史、长期记忆会不会污染判断、平台 ACK 和长任务 graph 怎么解耦、AI 输出什么时候必须人工确认。每个问题最后都变成了一个架构决策，比如结构化上下文、检索式记忆、ReviewGate、HumanApproval 和 outbox。这体现的是把模糊问题拆成具体机制的能力。

### 如果问：你和 HR 团队协作时能做什么？

可以答：

> 我可以先把 HR 的业务语言翻译成技术团队能实现的需求文档，比如场景目标、字段定义、流程节点、异常分支、验收标准和风险边界；同时也能把技术限制解释给 HR 团队，比如哪些结论需要人工确认，哪些数据需要结构化，哪些效果需要通过复盘指标观察。我希望做的是技术和业务之间的连接者。
