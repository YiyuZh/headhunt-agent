# 故障排查手册

## 状态

当前已有工程骨架、数据库基础、历史 Feishu outbox 和内部 graph/approval 测试路径。PRD 已切换为 Discord First；Discord interaction、button、modal、War Room thread/message 尚未实现，真实 Discord 联调未验证。本手册记录目标第一版完成后的排查路径。

## Discord Endpoint 保存失败

现象：
- Discord Developer Portal 提示 Interactions Endpoint URL 无效。

排查：
1. 确认公网 HTTPS URL 指向本机服务。
2. 确认路径是 `/discord/interactions`。
3. 查看服务日志是否收到 PING。
4. 确认 `DISCORD_PUBLIC_KEY` 正确。
5. 确认服务用 raw body 校验 `X-Signature-Ed25519` 和 `X-Signature-Timestamp`。
6. 确认 PING 响应没有同步跑 graph 或数据库慢操作。

## Slash Command 无响应

现象：
- 用户输入 `/headhunt new` 后 Discord 显示应用未响应。

排查：
1. 确认 command 已注册到当前 guild。
2. 确认 Bot 在 guild/channel 中。
3. 确认 `DISCORD_ALLOWED_GUILD_IDS` 和 `DISCORD_ALLOWED_CHANNEL_IDS` 包含当前位置。
4. 检查 `/discord/interactions` 是否 3 秒内 ACK/defer。
5. 检查 `discord_event_logs` 是否写入。
6. 检查 `discord_outbox` 是否有 `task_intake`。

## Double Check 后 Graph 没启动

现象：
- 用户点了 Approve，但没有 War Room 进度。

排查：
1. 检查 `discord_interactions` 是否记录 `task_check:approve`。
2. 检查 `TaskDoubleCheckState.status` 是否为 `approved`。
3. 检查 `discord_outbox` 是否有 `graph_dispatch`。
4. 确认 worker 正在运行。
5. 检查 outbox 是否被 claim 后失败或进入 dead_letter。
6. 查看 `graph_threads` 和 `agent_runs`。
7. 检查 LangGraph checkpointer 是否连接 PostgreSQL：
8. 确认 Approve 时已冻结 `CanonicalTaskBrief.version`，且 `graph_dispatch` 使用的是冻结版本。

```bash
python -m app.runtime.postgres_smoke
```

如果只想排查主库和迁移状态，先跳过 checkpointer setup：

```bash
python -m app.runtime.postgres_smoke --skip-checkpointer-setup
```

## Double Check 字段来源缺失

现象：
- 确认卡里出现无来源的岗位事实、候选人事实或公司当前事实。

排查：
1. 检查 `TaskIntakeParser` 输出的每个关键字段是否有 `field_source` 和 `confidence`。
2. 无 `field_source` 的内容必须进入 `assumptions`，不能进入 `facts`。
3. 检查 Discord Edit modal 是否生成了新的 `CanonicalTaskBrief.version`。
4. 检查 Approve 后版本是否冻结，下游 Agent 是否只读冻结版。
5. 如果字段来自公司背调、在职状态、融资、新闻、裁员或组织变化，必须检查是否有 `SearchGateway source_refs`。

## ACK 很快，但没有后续进度卡

这是正常架构的一半：ACK 只代表事件已收到和入队，不代表 graph 完成。

继续排查：
1. 查看 `discord_outbox` pending / failed / dead_letter。
2. 确认 worker 正在运行。
3. 检查 Discord message send/update 权限。
4. 检查 bot token 是否有效。
5. 检查限频响应和重试时间。

## 审批按钮无效

现象：
- 用户点击 approve / reject / edit 后没有继续执行。

排查：
1. 检查 button/modal custom_id 是否包含 `thread_id`、`action_id`、`interrupt_id`、`idempotency_key`。
2. 检查 signature 校验是否通过。
3. 检查 `discord_interactions.idempotency_key` 是否重复。
4. 检查是否存在 pending `ActionProposal`。
5. 检查 `HumanApproval` 是否写入。
6. 检查 `discord_outbox(kind='resume')` 是否入队。
7. `edit` 必须带 `edited_payload`，否则应拒绝。

## 重复执行

现象：
- 重复创建任务、重复 resume、重复发送消息。

排查：
1. 检查 `discord_event_logs.interaction_id` 唯一约束。
2. 检查 `discord_outbox.idempotency_key` 唯一约束。
3. 检查 `discord_interactions.idempotency_key` 唯一约束。
4. 检查 worker claim 是否原子。
5. 检查 `ActionProposal` 和 `HumanApproval` 的 `idempotency_key`。

## token 消耗异常

现象：
- 单次任务 token 过高，速度慢。

排查：
1. 查看 AgentRuns 的 `token_estimate`。
2. 查看 ContextPack 的 `excluded_context_reason`。
3. 检查是否误把完整历史、完整 state、全量 artifacts 或全量长期记忆注入。
4. 检查 `max_memory_items` 和 `max_context_tokens`。
5. 检查 `MemoryGateway.retrieve` 是否执行 MMR 和预算裁剪。
6. 检查 SOPRegistry 是否注入了全部 SOP，而不是 1 个主 SOP + 最多 2 个审查 SOP。

## ReviewGate 反复失败

现象：
- 产物一直回退修复，无法进入下一步。

排查：
1. 查看 `ReviewResult.status`。
2. 查看 `failed_fields` 和 `fix_suggestions`。
3. 确认 ReviewGate 是 artifact-level quality gate，只审查当前产物，不是全局聊天式审查。
4. 确认 graph 使用 conditional edge 路由 `pass / needs_fix / needs_human`。
5. `needs_fix` 只能回到当前 artifact 对应的 `repair_node` 一次，不能重跑完整 graph 或无关上游节点。
6. 如果第二次仍 `needs_fix`，应升级为 `interrupt_human_approval`，不能无限自动重试。
7. 检查 repair retry 计数是否绑定 `artifact_id + reviewer_id + version`，避免换节点后绕过“一次修复”限制。
8. SchemaValidator 失败时优先检查输出 schema。
9. PracticalityReviewer 失败时检查话术是否有个性化、价值主张和明确行动请求。
10. ContextBudgetReviewer 失败时检查是否传入全历史或过量 memory。

## 记忆检索不准确

现象：
- 命中了明显无关记忆，或该命中的记忆没命中。

排查：
1. 查看 `memory_retrieval_audits`。
2. 检查 scope filter 是否按 `thread_id / tenant_id / guild_id / user_id / project_id / requisition_id / candidate_id` 执行，且没有跨 thread、跨租户、跨 guild、跨项目命中。
3. 检查 memory 是否 `status=active`。
4. 检查 `expires_at` 是否已过期。
5. 检查 `retention_policy` 是 `30d`、`90d` 还是 `permanent`。
6. 检查 embedding 模型和维度是否一致。
7. 检查 rerank、recency、source trust、retention weighting 和 MMR 参数。
8. 检查返回给 Agent 的是否只有 `MemoryRef` 摘要和 ref，而不是长期记忆全文。

## 当前事实或关键词看起来不对

现象：
- 公司事实过期，或简历关键词明显偏离当前简历/JD。

排查：
1. 公司背调、当前在职、融资、新闻、裁员、组织变化必须检查 `SearchGateway source_refs`。
2. 长期记忆不得替代实时搜索；它只能提供 taxonomy、项目规则、用户修正、历史案例和 SOP。
3. 简历关键词抽取应优先读取当前简历 artifact 和当前 JD artifact。
4. 如果关键词来自长期记忆，检查它是否只作为 taxonomy 或用户修正规则使用，而不是覆盖当前简历/JD。
5. 检查 `ContextPack.artifact_refs` 是否包含本次任务的简历/JD artifact，且没有被旧 artifact 误替代。

## Deferred Feishu Adapter 故障

如果用户明确启用飞书 adapter，再按旧路径排查：

```text
/feishu/events URL challenge
/feishu/card-actions 卡片回传
feishu_outbox
FeishuGateway token
Bitable app_token/table_id
```

该路径不再作为第一版主链路验收 blocker。
