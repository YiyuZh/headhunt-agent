# 故障排查手册

本文按 Feishu First 第一版主链路排查。Discord 仅作为 optional adapter。

## 飞书回调保存失败

现象：

```text
飞书开放平台提示事件订阅 URL 无法保存
返回数据不是合法的JSON格式
```

检查：

1. 域名必须是 HTTPS。
2. 路径必须是 `https://<DOMAIN>/feishu/events`。
3. `FEISHU_VERIFICATION_TOKEN` 和 `FEISHU_ENCRYPT_KEY` 是否与后台一致。
4. 服务日志里是否有 challenge 请求。
5. `/feishu/events` 是否在 3 秒内返回 challenge。

先不要继续点飞书后台“保存”。在服务器上先确认 HeadHunt HTTPS 和 API 真的通：

```bash
curl -i https://headhunt.zenithy.art/health
```

必须看到 `HTTP/2 200` 或 `HTTP/1.1 200`，并且响应体是 JSON，例如：

```json
{"status":"ok"}
```

再手动模拟飞书 challenge：

```bash
cd /opt/apps/headhunt-agent
set -a
. ./.env
set +a

curl -i -sS -X POST "https://headhunt.zenithy.art/feishu/events" \
  -H "Content-Type: application/json" \
  -d "{\"challenge\":\"local-check\",\"token\":\"$FEISHU_VERIFICATION_TOKEN\",\"type\":\"url_verification\"}"
```

正确响应必须类似：

```http
HTTP/2 200
content-type: application/json

{"challenge":"local-check"}
```

如果飞书提示“返回数据不是合法的JSON格式”，按返回内容判断：

- 返回 `not found` 或 404：`/opt/apps/hiremate/Caddyfile` 里还没有 HeadHunt 站点块，或 Caddy 没 reload。
- 返回 502：`hiremate-caddy` 找不到 `lietou-api:8000`，通常是 HeadHunt compose 没启动、`api` 没进 `shared_gateway`、或上游容器名/网络不一致。
- 返回 HTML：请求打到了别的站点，常见是当前只有 `ttcintern` 配了 HTTPS，`headhunt.zenithy.art` 还没有正确接到统一网关。
- 返回 401 JSON：`FEISHU_VERIFICATION_TOKEN` 和飞书后台“加密策略”里的 token 不一致。
- 返回 503 JSON：运行时配置缺失，先按部署手册补 `.env` 并通过 `python3 -m app.runtime.local_doctor --strict`。

当前服务器状态要分清：已知只有 `ttcintern` 已配置 HTTPS；`headhunt.zenithy.art` 只有在 `curl -f https://headhunt.zenithy.art/health` 和上面的 challenge curl 都通过后，才能去飞书后台保存事件订阅。

## 卡片点击无响应

检查：

1. 卡片回调地址是否是 `https://<DOMAIN>/feishu/card-actions`。
2. 请求是否带 timestamp / nonce / signature。
3. `action.value` 是否包含 `thread_id/action_id/interrupt_id/idempotency_key/decision`。
4. `feishu_card_actions` 是否写入。
5. `feishu_outbox` 是否有 resume 任务。

## ACK 成功但 graph 没跑

ACK 只代表回调已收到并入队，不代表 graph 已完成。

检查：

```bash
docker compose logs worker
docker compose logs api
```

再检查：

- outbox 是否 pending / claimed / failed / dead_letter。
- worker 是否能连接 PostgreSQL。
- graph_dispatch 是否缺少 AgentHarness 或 runtime factory。

## Double Check 后没有进入正式 graph

检查：

1. 用户是否点了 approve，而不是 edit/reject。
2. `CanonicalTaskBrief.version` 是否已冻结。
3. `TaskDoubleCheckApproval` 是否写入。
4. `graph_dispatch` outbox 是否存在。

## Bitable 没有写入

这是可能的正确行为。Bitable 写入必须人工确认。

检查：

1. 是否存在 approved `ActionProposal`。
2. 是否存在 approved `HumanApproval`。
3. `feishu_outbox.kind='bitable_write'` 是否入队。
4. `client_token` 是否重复但 payload_hash 不同。
5. Bitable app_token/table_id/权限是否正确。

## 模型调用失败

检查：

- 用户是否通过飞书模型配置卡添加 chat profile。
- API Key 是否加密保存，list 不展示 key 是正确行为。
- OpenAI embedding profile 是否单独配置。
- DeepSeek chat profile 不可作为 embedding profile。

## Docker Hub 拉镜像失败

如果 `docker compose up -d --build` 报：

```text
Get "https://registry-1.docker.io/v2/": net/http: request canceled
```

这是 Docker Hub 拉镜像网络失败，不是 worker 业务逻辑失败，也不是 PyPI 下载慢。先在服务器执行：

```bash
docker pull python:3.11-slim
```

如果这一步也失败，先配置服务器 Docker daemon 的 `registry-mirrors` 或使用云厂商提供的 Docker Hub 镜像加速服务，然后：

```bash
sudo systemctl daemon-reload
sudo systemctl restart docker
docker pull python:3.11-slim
```

`PIP_INDEX_URL=https://pypi.tuna.tsinghua.edu.cn/simple` 只影响容器 build 里的 Python 包安装，不影响 Docker Hub 基础镜像拉取。

## source .env 时出现 `Headhunter: command not found`

如果执行：

```bash
set -a
. ./.env
set +a
```

出现：

```text
Headhunter: command not found
```

通常是 `.env` 里有带空格的值没有加引号，例如：

```env
APP_NAME=AI Headhunter Feishu War Room
```

改成：

```env
APP_NAME="AI Headhunter Feishu War Room"
```

然后重新加载 `.env`。这个错误不代表 API 挂了；如果 `/ready` 已返回 200，说明服务本身已经通。
