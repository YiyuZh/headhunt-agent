# 故障排查手册

本文按 Feishu First 第一版主链路排查。Discord 仅作为 optional adapter。

## 飞书回调保存失败

现象：

```text
飞书开放平台提示事件订阅 URL 无法保存
```

检查：

1. 域名必须是 HTTPS。
2. 路径必须是 `https://<DOMAIN>/feishu/events`。
3. `FEISHU_VERIFICATION_TOKEN` 和 `FEISHU_ENCRYPT_KEY` 是否与后台一致。
4. 服务日志里是否有 challenge 请求。
5. `/feishu/events` 是否在 3 秒内返回 challenge。

## 卡片点击无响应

检查：

1. 卡片回调地址是否是 `https://<DOMAIN>/feishu/card-actions`。
2. 请求是否带 timestamp / nonce / signature。
3. `action.value` 是否包含 `thread_id/action_id/interrupt_id/idempotency_key/decision`。
4. `feishu_card_actions` 是否写入。
5. `feishu_outbox` 是否有 resume 任务。

## ACK 成功但 graph 没跑

ACK 只代表回调已收到并入队，不代表 graph 已完成。

检查：

```bash
docker compose logs worker
docker compose logs api
```

再检查：

- outbox 是否 pending / claimed / failed / dead_letter。
- worker 是否能连接 PostgreSQL。
- graph_dispatch 是否缺少 AgentHarness 或 runtime factory。

## Double Check 后没有进入正式 graph

检查：

1. 用户是否点了 approve，而不是 edit/reject。
2. `CanonicalTaskBrief.version` 是否已冻结。
3. `TaskDoubleCheckApproval` 是否写入。
4. `graph_dispatch` outbox 是否存在。

## Bitable 没有写入

这是可能的正确行为。Bitable 写入必须人工确认。

检查：

1. 是否存在 approved `ActionProposal`。
2. 是否存在 approved `HumanApproval`。
3. `feishu_outbox.kind='bitable_write'` 是否入队。
4. `client_token` 是否重复但 payload_hash 不同。
5. Bitable app_token/table_id/权限是否正确。

## 模型调用失败

检查：

- 用户是否通过飞书模型配置卡添加 chat profile。
- API Key 是否加密保存，list 不展示 key 是正确行为。
- OpenAI embedding profile 是否单独配置。
- DeepSeek chat profile 不可作为 embedding profile。

## 未验证边界

- 真实飞书后台、真实卡片发送、真实 Bitable 权限、真实 OpenAI/DeepSeek、真实服务器 Docker Compose 联调未完成前，不得写成已通过。
