# 本地部署与使用手册

## 状态

当前项目已有 Phase 14 工程骨架：FastAPI / Pydantic、PostgreSQL + pgvector
SQLAlchemy models、冻结 Alembic 初始迁移、历史飞书事件/卡片回调 verifier、
durable outbox dispatcher、FeishuGateway、BitableGateway、ArtifactStore repository、
MemoryGateway / pgvector 检索、LangGraph 最小根图 / council 子图 / 业务子图和
PostgreSQL checkpointer factory 已生成；AgentHarness、OpenAI Responses LLMGateway、
OpenAI EmbeddingGateway、AgentRuns 写入、RunMemory 写入、War Room 卡片 outbox 和
ActionGate 提案已接入；业务路由已能创建 ActionProposal、触发真实 LangGraph
`interrupt()`、用 `Command(resume=HumanApproval)` 恢复同一 thread，并在 approve/edit
后把业务写入排入 durable `bitable_write` outbox。reject 不执行业务写入。
`lietou-outbox-worker` 已装配真实 worker 路径，用于领取并执行 durable outbox。
`lietou-postgres-smoke` 已装配真实 PostgreSQL/checkpointer smoke 检查入口。
`lietou-external-smoke` 已装配真实飞书/OpenAI 外部服务 smoke 检查入口，且
`POST /tasks/authorize` 已实现人工授权后排队 `graph_dispatch` 的入口，
`GET /threads/{thread_id}` 和 `GET /runs/{run_id}` 已实现只读审计查询入口，
`GET /approvals/{interrupt_id}` 和 `POST /approvals/{interrupt_id}/decision`
已实现内部人工确认入口。除 `/health` 和外部平台回调外，内部控制面 API 现在必须带
`X-Internal-Admin-Key`，生产/容器模式默认不暴露 `/docs`、`/redoc` 和
`/openapi.json`。

重要边界：当 runtime factory 注入真实 AgentHarness 时，`graph_dispatch` 可进入真实
harness 路径；`resume` 只有在 AgentHarness、ActionGate、ActionExecutor 都装配后才会
生产放行。当前已用内存 checkpointer 验证 interrupt/resume 行为；真实 PostgreSQL
checkpointer 端到端仍未联调。

本轮 PRD 已切换为 Discord First；Discord interaction、TaskIntakeParser double check、
DiscordGateway、ReviewGate、AgentSOPRegistry 和 memory retention job 尚未实现，真实
Discord 联调未验证。飞书/Bitable 只作为 deferred adapter。

当前已补齐 Docker Compose 部署底座：服务器在项目根目录复制 `.env` 后执行
`docker compose up -d --build`，会启动 FastAPI API、durable outbox worker、
PostgreSQL + pgvector 和 Caddy HTTPS 反向代理。Discord 业务入口代码仍未实现，
所以 Discord 真实联调仍写“未验证”。Caddy 默认只公开 `/health`、未来
`/discord/interactions`、历史 `/feishu/events` 和 `/feishu/card-actions`；内部路径
需要 admin key。

说明：
- Docker Compose 内部数据库 host 是 `postgres`。
- `POSTGRES_BIND=127.0.0.1` 表示 PostgreSQL 端口只绑定服务器本机，避免默认暴露到公网。
- Docker Compose 默认会从 `POSTGRES_USER`、`POSTGRES_PASSWORD`、`POSTGRES_HOST`、`POSTGRES_INTERNAL_PORT` 和 `POSTGRES_DB` 派生 `DATABASE_URL` 和 `CHECKPOINT_DB_URL`，不要在 `.env` 里重复手写，避免密码不一致。
- `POSTGRES_PASSWORD` 和 `INTERNAL_ADMIN_API_KEY` 必须显式填写；Compose 不再提供弱默认值。
- 容器连接外部数据库时，优先改 `POSTGRES_HOST` / `POSTGRES_INTERNAL_PORT` / `POSTGRES_USER` / `POSTGRES_PASSWORD` / `POSTGRES_DB`。
- 只有在宿主机直接运行 Python 时，才手动设置 `DATABASE_URL` 和 `CHECKPOINT_DB_URL`。
- 服务器部署时 `DOMAIN` 必须填写已经解析到服务器的域名，并在 Discord Developer Portal 使用 `https://<DOMAIN>/discord/interactions`。

### 服务器 Docker Compose 部署

在服务器安装 Docker Engine / Docker Compose v2 后：


## 本地 Discord 联调

真实 Discord 联调要等 `/discord/interactions`、DiscordGateway、command 注册和
discord_outbox 实现后再执行。当前尚未实现，以下是目标流程。服务器 Docker Compose
部署时不需要 ngrok，直接使用 Caddy 提供的公网 HTTPS 域名。

1. 确认域名解析到服务器。
2. 执行 `docker compose up -d --build`。
3. 把公网地址填入 Discord Developer Portal：

```text
Interactions Endpoint URL: https://your-domain/discord/interactions
```

4. Discord 保存 endpoint 时会发 PING，系统必须校验 signature 并返回 PONG。
5. 注册 `/headhunt new`、`/headhunt candidate`、`/headhunt map`、`/headhunt status`。
6. 在允许的 channel 使用 slash command 触发任务。
7. 先确认任务 double check 卡，再进入正式 graph。
8. double check 卡必须展示关键字段的 `field_source`、`confidence` 和 `assumptions`。
9. 用户 Approve 后冻结 `CanonicalTaskBrief.version`；Edit 必须生成新版本并再次确认。
10. graph_dispatch 只能读取冻结版 `CanonicalTaskBrief`、ArtifactRef、MemoryRef、SOPRef 和 SourceRef。

## 本地飞书联调（Deferred Adapter）

真实飞书联调要等用户明确启用飞书 adapter、真实飞书应用、HTTPS 回调地址和真实
PostgreSQL 服务后再执行。当前历史 Feishu verifier、outbox、Bitable gateway 和相关
测试可复用，但它不再是第一版主入口。

## 常用 API 调用

### 运行会审

```bash
curl -X POST http://127.0.0.1:8000/council/deliberate \
  -H "Content-Type: application/json" \
  -H "X-Internal-Admin-Key: <INTERNAL_ADMIN_API_KEY>" \
  -d '{
    "request_text": "帮我做一个 AI 平台后端负责人岗位的人才地图",
    "source": "manual"
  }'
```

返回中必须看到：
- `council_mode`
- `mode_reason`
- `required_agents`
- `optional_agents`

### 授权任务

```bash
curl -X POST http://127.0.0.1:8000/tasks/authorize \
  -H "Content-Type: application/json" \
  -H "X-Internal-Admin-Key: <INTERNAL_ADMIN_API_KEY>" \
  -d '{
    "request_text": "请用三省六部完整会审 AI 平台负责人岗位并生成人才地图",
    "source": "manual",
    "source_ref": "manual-20260604-001",
    "approved": true,
    "approver": {
      "source": "manual",
      "user": "operator"
    }
  }'
```

返回中重点看：

- `status`: 应为 `queued`。
- `source_ref`: 本次任务的幂等身份。你显式传入时原样返回；未传时系统生成
  `request:<hash>`。
- `council_mode` / `mode_reason`: 当前自动选择的会审模式和理由；输入包含“三省六部”
  时必须返回 `full_council`。
- `idempotency_key`: 写入 outbox 的幂等键。重复提交同一 `source/source_ref` 不应重复排队。
- `outbox_payload_ref`: 授权 payload 的 ArtifactBlob 引用。

注意：这个接口只表示“人工同意开始任务，并把 graph_dispatch 放进 durable outbox”。
它不会在 HTTP 请求里同步跑完整 graph，也不会写业务数据。启动
`lietou-outbox-worker` 后，worker 才会异步领取 outbox 并进入
`headhunter_war_room_graph`。

注意：`/tasks/authorize` 是当前 Discord 未实现前的内部调试入口，必须显式传
`approved=true` 才会模拟“已人工授权”的状态；请求默认 `approved=false`，不会排队
graph。正式 Discord First 路径必须先经过 `TaskIntakeParser -> double check
confirmation card -> Approve/Edit/Reject`；未 Approve 的 `CanonicalTaskBrief` 不得进入正式 graph。

### 查看线程和运行审计

```bash
curl -H "X-Internal-Admin-Key: <INTERNAL_ADMIN_API_KEY>" \
  http://127.0.0.1:8000/threads/<thread_id>
```

返回中可查看：

- `status` / `state_summary`：线程当前状态。
- `artifact_refs`：只读 artifact 摘要和 `content_ref`，不展开全文。
- `memory_refs`：本线程最近运行命中的记忆引用。
- `pending_interrupts`：等待人工确认的动作。
- `recent_runs`：最近 AgentRun 摘要。

查看某次 AgentRun：

```bash
curl -H "X-Internal-Admin-Key: <INTERNAL_ADMIN_API_KEY>" \
  http://127.0.0.1:8000/runs/<run_id>
```

该接口只返回 `context_pack_ref`、输入/输出摘要、refs、token 估算和状态，不返回原始
prompt 或完整 ContextPack。

### 查看和提交人工确认

当 `/threads/<thread_id>` 返回 `pending_interrupts` 后，可以复制其中的
`interrupt_id` 查看待确认动作：

```bash
curl -H "X-Internal-Admin-Key: <INTERNAL_ADMIN_API_KEY>" \
  http://127.0.0.1:8000/approvals/<interrupt_id>
```

返回中重点看：

- `action_id` / `thread_id` / `interrupt_id`：resume 必须匹配的三元身份。
- `payload_summary` / `payload_ref`：待确认动作摘要和草稿引用；接口不会展开全文。
- `status` / `can_decide`：是否仍可提交确认。

批准执行：

```bash
curl -X POST http://127.0.0.1:8000/approvals/<interrupt_id>/decision \
  -H "Content-Type: application/json" \
  -H "X-Internal-Admin-Key: <INTERNAL_ADMIN_API_KEY>" \
  -d '{
    "decision": "approve",
    "approver": {
      "source": "internal",
      "user": "operator"
    }
  }'
```

拒绝执行：

```bash
curl -X POST http://127.0.0.1:8000/approvals/<interrupt_id>/decision \
  -H "Content-Type: application/json" \
  -H "X-Internal-Admin-Key: <INTERNAL_ADMIN_API_KEY>" \
  -d '{
    "decision": "reject",
    "approver": {
      "source": "internal",
      "user": "operator"
    }
  }'
```

修改后执行：

```bash
curl -X POST http://127.0.0.1:8000/approvals/<interrupt_id>/decision \
  -H "Content-Type: application/json" \
  -H "X-Internal-Admin-Key: <INTERNAL_ADMIN_API_KEY>" \
  -d '{
    "decision": "edit",
    "approver": {
      "source": "internal",
      "user": "operator"
    },
    "edited_payload": {
      "note": "人工修正后的执行说明"
    }
  }'
```

注意：该接口只写 `HumanApproval` 和 durable `resume` outbox，不直接写业务数据。
启动 `lietou-outbox-worker` 后，worker 才会用 `Command(resume=HumanApproval)` 恢复
graph，并由 ActionExecutor 再次校验后排队数据库写入、报告发布、外部触达或 deferred
Bitable 等业务副作用。

## 第一版目标完成后你可以做什么

- 在 Discord button / modal 里 approve / reject / edit。
- double check 系统解析出来的结构化任务。
- 导入候选人或历史案例。
- 让系统筛选候选人与岗位匹配证据。
- 审批长期记忆是否进入可检索池。
- 检查任务确认卡里的 `field_source`、`confidence` 和 `assumptions`。
- 检查 ReviewGate 对当前 artifact 的 `pass / needs_fix / needs_human` 结果。
- 检查公司当前事实是否有 SearchGateway source_refs，简历关键词是否来自当前简历/JD artifact。

## 人工必须做什么

- 创建 Discord Application 和 Bot。
- 填写 `.env` 中的 Discord Public Key、Bot Token、Application ID、guild/channel allowlist。
- 配置 Interactions Endpoint URL。
- 注册 slash commands。
- 把 Bot 邀请到测试 Discord Server。
- 为业务写入、报告发布、任务创建、外部触达和候选人推荐结论做确认。
- 对 double check 结果执行 Approve/Edit/Reject；Approve 后版本冻结，改动必须走 Edit 生成新版本。
- 对第二次仍 `needs_fix` 的 artifact-level ReviewGate 结果做人工确认，不能让系统无限修复或重跑完整 graph。
- 对长期记忆 active、续期或 permanent 升级做人工审批。
