# 服务器部署与使用手册

本文是 `headhunt-agent` 的云服务器本土化上线 runbook。目标是让你在一台 Linux 服务器上，把 FastAPI API、worker、PostgreSQL + pgvector、统一 Caddy 网关、飞书事件/卡片回调先跑通；Bitable 同步作为第二阶段业务同步单独验证。

当前服务器实际状态要分清楚：已知只有 `ttcintern` 站点完成了 HTTPS 配置。`headhunt.zenithy.art` 是本项目目标域名，必须等 `/opt/apps/hiremate/Caddyfile` 增加 HeadHunt 站点块、Caddy validate/reload 成功、DNS 生效、并且 `curl -f https://headhunt.zenithy.art/health` 通过后，才算 HeadHunt 的 HTTPS 已配置完成。

HeadHunt 的目标部署口径是统一网关模式：`/opt/apps/hiremate` 里的 `hiremate-caddy` 是唯一公网 `80/443` 入口，`headhunt-agent` 只启动 `lietou-api`、`lietou-worker` 和 `lietou-postgres`，并通过 Docker 网络 `shared_gateway` 让统一 Caddy 反代到 `lietou-api:8000`。服务器上直接在 `/opt/apps/headhunt-agent` 执行 `docker compose up -d --build`，不要让本项目自带 `lietou-caddy` 抢占公共端口。

## 2. 服务器准备

服务器建议使用 Ubuntu 22.04/24.04。先准备：

- Docker Engine 和 Docker Compose v2。
- `git`、`openssh-client`、`python3`、`curl`、`nano`。
- 云安全组/防火墙放行 `80` 和 `443`。
- DNS A 记录：`headhunt.zenithy.art -> 140.143.119.81`。
- 一个飞书开放平台应用和测试群；Bitable app 只在第二阶段业务同步时需要。

检查基础命令：

```bash
docker compose version
git --version
python3 --version
curl --version
```

检查 DNS：

```bash
dig +short headhunt.zenithy.art
```

输出应是 `140.143.119.81`。如果没有 `dig`，可以用：

```bash
getent hosts headhunt.zenithy.art
```

## 3. 配置 GitHub Deploy Key

服务器上推荐使用项目专用 Deploy Key。不要把仓库 SSH 地址当成 Deploy Key 内容；Deploy Key 内容是 `.pub` 公钥文件里的整行文本。

生成项目专用 key：

```bash
mkdir -p ~/.ssh
chmod 700 ~/.ssh
ssh-keygen -t ed25519 -C "deploy-headhunt-agent" -f ~/.ssh/id_ed25519_headhunt_agent
cat ~/.ssh/id_ed25519_headhunt_agent.pub
```

打开 GitHub 仓库：

```text
YiyuZh/headhunt-agent -> Settings -> Deploy keys -> Add deploy key
```

把 `cat` 输出的 `.pub` 公钥整行粘进去。只部署代码时不需要勾选 write access。

然后在服务器配置项目级 SSH alias，避免覆盖多项目服务器上的全局 `Host github.com`：

```bash
nano ~/.ssh/config
```

加入：

```sshconfig
Host github.com-headhunt-agent
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519_headhunt_agent
    IdentitiesOnly yes
```

设置权限并测试：

```bash
chmod 600 ~/.ssh/config ~/.ssh/id_ed25519_headhunt_agent
ssh -T github.com-headhunt-agent
```

如果看到类似下面的提示，这是成功，不是错误：

```text
Hi YiyuZh/headhunt-agent! You've successfully authenticated, but GitHub does not provide shell access.
```

再确认能读取仓库：

```bash
git ls-remote git@github.com-headhunt-agent:YiyuZh/headhunt-agent.git refs/heads/main
```

## 5. 生成并填写 `.env`

在项目根目录执行：

```bash
cd /opt/apps/headhunt-agent
bash scripts/lietou-oneclick.sh
```

这一步会：

- 如果 `.env` 不存在，从 `.env.example` 创建。
- 自动生成 `INTERNAL_ADMIN_API_KEY`、`POSTGRES_PASSWORD`、`MODEL_SECRET_ENCRYPTION_KEY`。
- 检查 Docker、Compose、必填环境变量和占位值。
- 不启动服务，不输出密钥值。

编辑 `.env`：

```bash
nano .env
```

至少确认这些值：

```env
DOMAIN=headhunt.zenithy.art
HTTP_PORT=80
HTTPS_PORT=443
DATABASE_WAIT_SECONDS=60

PIP_INDEX_URL=https://pypi.tuna.tsinghua.edu.cn/simple
PIP_DEFAULT_TIMEOUT=120
PIP_RETRIES=5

INTERNAL_ADMIN_API_KEY=<auto-generated>
MODEL_SECRET_ENCRYPTION_KEY=<auto-generated>

POSTGRES_DB=lietou
POSTGRES_USER=lietou
POSTGRES_PASSWORD=<auto-generated>
POSTGRES_HOST=postgres
POSTGRES_INTERNAL_PORT=5432
POSTGRES_BIND=127.0.0.1
POSTGRES_PORT=5432

CHANNEL_GATEWAY_PROVIDER=feishu
FEISHU_BASE_URL=https://open.feishu.cn
FEISHU_APP_ID=
FEISHU_APP_SECRET=
FEISHU_VERIFICATION_TOKEN=
FEISHU_ENCRYPT_KEY=
FEISHU_DEFAULT_CHAT_ID=
FEISHU_BITABLE_APP_TOKEN=
FEISHU_BITABLE_REQUISITION_TABLE_ID=
FEISHU_BITABLE_CANDIDATE_TABLE_ID=
FEISHU_BITABLE_TALENT_MAP_TABLE_ID=
FEISHU_BITABLE_REPORT_TABLE_ID=

MODEL_PROVIDER_ALLOWLIST=openai,deepseek
EMBEDDING_PROVIDER=openai
EMBEDDING_MODEL=text-embedding-3-small
EMBEDDING_API_KEY=
```

填写规则：

- `DOMAIN` 固定填 `headhunt.zenithy.art`，不要填 `localhost`。
- `PIP_INDEX_URL` 默认使用清华 PyPI 镜像，适合国内云服务器加速 `docker compose up -d --build`。如果服务器在海外或清华源异常，可临时改成 `https://pypi.org/simple` 后重新 build。
- `INTERNAL_ADMIN_API_KEY`、`POSTGRES_PASSWORD`、`MODEL_SECRET_ENCRYPTION_KEY` 这三项不要手填 `<auto-generated>` 字面量。它们应当是 `bash scripts/lietou-oneclick.sh` 写入 `.env` 的真实随机字符串；如果 `.env` 里仍然看到 `<auto-generated>`、`change-this-*` 或空值，重新执行 `bash scripts/lietou-oneclick.sh`。
- `POSTGRES_BIND=127.0.0.1` 表示 PostgreSQL 只暴露给服务器本机，生产环境不要改成 `0.0.0.0`。
- 用户业务模型 Key 不写进共享 `.env`。第一版主路径是用户在飞书模型配置卡里 BYOK。
- `EMBEDDING_API_KEY` 是可选的本地 embedding fallback。第一版可以先留空；留空时服务仍可启动，但长期记忆向量检索会显示未就绪。只有当你想让服务器直接使用 OpenAI embedding fallback 时，才填写一个 OpenAI API Key。
- `FEISHU_APP_ID`、`FEISHU_APP_SECRET`、`FEISHU_VERIFICATION_TOKEN`、`FEISHU_ENCRYPT_KEY`、`FEISHU_DEFAULT_CHAT_ID` 是第一阶段飞书接入必填项，获取方式见 [飞书接入操作手册](飞书接入操作手册.md)。不要把群名称当成 `chat_id`。
- `FEISHU_BITABLE_APP_TOKEN` 和 4 个 `FEISHU_BITABLE_*_TABLE_ID` 是第二阶段 Bitable 同步项；可以先留空，strict preflight 不再因此失败，但审批后的多维表格同步会显示未就绪。不要把 Bitable 表名、view ID 或 record ID 当成这些字段。
- 如果你在飞书后台看到“事件配置 / 回调配置 / 加密策略”，先按 [飞书接入操作手册](飞书接入操作手册.md) 的“先按飞书后台配置事件与回调”执行：事件订阅选“将事件发送至开发者服务器”，事件 URL 填 `/feishu/events`，卡片回调填 `/feishu/card-actions`，再从“加密策略”复制 `Verification Token` 和 `Encrypt Key`。

如果需要确认自动生成是否完成，只看变量名，不要把值发到聊天里：

```bash
grep -E '^(INTERNAL_ADMIN_API_KEY|POSTGRES_PASSWORD|MODEL_SECRET_ENCRYPTION_KEY|EMBEDDING_API_KEY)=' .env
```

只想预检、不创建 `.env` 时用：

```bash
bash scripts/lietou-oneclick.sh --no-init --doctor-only
```

到这里先不要跑 `python3 -m app.runtime.local_doctor --strict`。先按飞书手册开通权限、发布版本、获取 `chat_id`，再把第一阶段飞书字段填进 `.env`。Bitable 可以第二阶段再填。

## 6. 统一 Caddy 网关

当前服务器采用统一网关模式，`hiremate-caddy` 是唯一公网 `80/443` 入口。注意区分“目标配置”和“已生效配置”：当前已知只有 `ttcintern` 完成 HTTPS；HeadHunt 需要在统一网关里新增站点块并重载后才会有 HTTPS。

`headhunt-agent` 的默认 `docker-compose.yml` 已经按目标网关模式配置：

- `api` 同时加入项目默认网络和 `shared_gateway`，默认网络访问 `postgres`，共享网络给统一 Caddy 访问。
- `api` 在 `shared_gateway` 上有 `lietou-api` 和 `headhunt-api` 两个别名。
- 项目自带 `lietou-caddy` 已放入 `standalone` profile，普通 `docker compose up -d --build` 不会启动它。
- 目标状态是：`hiremate` 的统一网关 Caddyfile 里，`headhunt.zenithy.art` 反代到 `lietou-api:8000`。

服务器路径固定为：

```text
统一网关: /opt/apps/hiremate
HeadHunt: /opt/apps/headhunt-agent
```

`/opt/apps/hiremate/Caddyfile` 中如果还没有 HeadHunt 站点块，就按下面口径新增；如果已经新增，则保留这些路由口径：

```text
公开路由: /health, /discord/interactions, /feishu/events, /feishu/card-actions
内部路由: /ready, /version, /tasks*, /approvals*, /threads*, /runs*, /council*, /docs*, /redoc*, /openapi.json
内部保护: 缺少 X-Internal-Admin-Key 时直接 403
上游服务: lietou-api:8000
其它路径: 404
```

首次改统一网关前先备份 `hiremate`：

```bash
cd /opt/apps/hiremate
RELEASE_ID="$(date +%Y%m%d-%H%M%S)"
mkdir -p /opt/apps/backups/releases/$RELEASE_ID/hiremate
cp Caddyfile /opt/apps/backups/releases/$RELEASE_ID/hiremate/Caddyfile.before
docker compose config > /opt/apps/backups/releases/$RELEASE_ID/hiremate/docker-compose.rendered.yml
```

确认网关容器和共享网络：

```bash
cd /opt/apps/hiremate
docker compose ps caddy
docker network inspect shared_gateway
```

如果 `shared_gateway` 不存在，先在网关目录按当前 `hiremate` 手册或 compose 口径创建/启动统一网关；完成这一步后，HeadHunt 目录的日常启动只需要 `docker compose up -d --build`。

修改 `/opt/apps/hiremate/Caddyfile` 后重载统一 Caddy：

```bash
cd /opt/apps/hiremate
docker compose exec caddy caddy validate --config /etc/caddy/Caddyfile
docker compose exec caddy caddy reload --config /etc/caddy/Caddyfile
```

只有上面两条命令成功，并且 `curl -f https://headhunt.zenithy.art/health` 返回成功后，才能说 HeadHunt 的 HTTPS 已配置。否则只能说 `ttcintern` 已有 HTTPS，HeadHunt 仍处于待配置/待验证。

只有在没有统一网关、需要单项目直连公网时，才使用 standalone 模式启动项目自带 Caddy：

```bash
cd /opt/apps/headhunt-agent
docker compose --profile standalone up -d --build
```

standalone 模式会启动 `lietou-caddy` 并监听服务器 `80/443`，不要和 `hiremate-caddy` 同时使用。

## 7. 上线前备份

每次上线或大改 `.env`、Caddyfile、Compose 前先做 release 备份：

```bash
cd /opt/apps/headhunt-agent
RELEASE_ID="$(date +%Y%m%d-%H%M%S)-$(git rev-parse --short HEAD)"
BACKUP_DIR="/opt/apps/backups/releases/$RELEASE_ID/headhunt-agent"
mkdir -p "$BACKUP_DIR"

git rev-parse HEAD > "$BACKUP_DIR/git-head.txt"
git status --short > "$BACKUP_DIR/git-status.txt"
cp .env "$BACKUP_DIR/.env" 2>/dev/null || true
cp Caddyfile "$BACKUP_DIR/Caddyfile.project" 2>/dev/null || true
docker compose config > "$BACKUP_DIR/docker-compose.rendered.yml"
```

不要执行 `docker compose down -v`。`-v` 会删除数据卷，可能直接清空 PostgreSQL 数据。

## 8. 启动前严格预检

先确认第 5 节的第一阶段飞书字段已经填完，再运行：

```bash
cd /opt/apps/headhunt-agent
python3 -m app.runtime.local_doctor --strict
```

如果 strict 报 `feishu_required still placeholder`，不要反复重跑这条命令。先回到第 5 节，把飞书后台的 `App ID`、`App Secret`、`Verification Token`、`Encrypt Key` 和群 `chat_id` 填进 `.env`。

必须看到 `status` 为 `ok` 才继续。常见 blocker：

- `runtime_required still placeholder`：本地密钥没有生成，重新执行 `bash scripts/lietou-oneclick.sh`。
- `feishu_required missing`：第一阶段飞书必填项没填。
- `feishu_required still placeholder`：第一阶段飞书字段仍是空值、`change-this-*` 或示例占位，按飞书手册补齐后再检查。
- `Optional Bitable business sync is not ready`：Bitable 第二阶段同步项没填；这不阻断服务启动、事件订阅、卡片回调和群消息。
- `Install Docker Engine/Compose`：Docker 或 Compose v2 不可用。

还可以先检查 Compose 插值：

```bash
docker compose config --quiet
```

## 9. 直接启动

```bash
cd /opt/apps/headhunt-agent
docker compose up -d --build
```

这就是服务器上 HeadHunt 的默认启动命令。它会启动 API、worker 和 PostgreSQL，并把 API 接入 `shared_gateway`；不会启动项目自带 `lietou-caddy`，所以不会抢 `80/443`。

如果想通过脚本把 strict preflight 和启动串起来，也可以用：

```bash
bash scripts/lietou-oneclick.sh --start
```

如果只想复用已有镜像，不重新 build：

```bash
docker compose up -d
```

## 10. 启动后验收

查看容器：

```bash
docker compose ps
```

期望看到 `lietou-api`、`lietou-worker`、`lietou-postgres` 是 running/healthy。统一网关模式下不应该看到 `lietou-caddy` 运行；公网入口应该是 `/opt/apps/hiremate` 里的 `hiremate-caddy`。

查看日志：

```bash
docker compose logs -f api worker postgres
cd /opt/apps/hiremate
docker compose logs -f caddy
```

检查 HTTP/HTTPS：

```bash
curl -f https://headhunt.zenithy.art/health
curl -f -H "X-Internal-Admin-Key: <INTERNAL_ADMIN_API_KEY>" https://headhunt.zenithy.art/ready
```

如果这里失败，不要去飞书后台保存 `https://headhunt.zenithy.art/feishu/events`。先回到第 6 节完成统一网关的 HeadHunt 站点块、Caddy validate/reload 和 DNS 检查。

容器内配置、数据库和记忆 smoke：

```bash
docker compose exec api lietou-config-check --strict
docker compose exec api lietou-postgres-smoke
docker compose exec api lietou-memory-retention --dry-run
```

如果 `lietou-config-check --strict` 返回 `status=degraded`，但唯一问题是 `feishu_bitable` / `Optional Bitable business sync is not ready`，第一阶段飞书事件、卡片回调和群消息可以继续联调；等第二阶段启用 Bitable 时再补多维表格权限和 `.env`。

单次 worker 检查：

```bash
docker compose exec worker lietou-outbox-worker --once
```

## 11. 飞书后台配置

在飞书开放平台配置。详细点击路径见 [飞书接入操作手册](飞书接入操作手册.md)：

保存前先确认服务器返回 JSON challenge：

```bash
cd /opt/apps/headhunt-agent
set -a
. ./.env
set +a

curl -i -sS -X POST "https://headhunt.zenithy.art/feishu/events" \
  -H "Content-Type: application/json" \
  -d "{\"challenge\":\"local-check\",\"token\":\"$FEISHU_VERIFICATION_TOKEN\",\"type\":\"url_verification\"}"
```

必须看到 `{"challenge":"local-check"}`。如果飞书后台提示“返回数据不是合法的JSON格式”，说明当前 URL 没有打到 FastAPI 的 `/feishu/events`，先按 [故障排查手册](故障排查手册.md) 的“飞书回调保存失败”排查 Caddy/DNS/API。

```text
事件订阅 URL: https://headhunt.zenithy.art/feishu/events
卡片回调 URL: https://headhunt.zenithy.art/feishu/card-actions
```

注意：

- “事件配置”的订阅方式要选“将事件发送至开发者服务器”，不是“长连接”。
- “加密策略”里的 `Verification Token` 和 `Encrypt Key` 要复制到服务器 `.env`。
- 两个回调入口使用同一组 `Verification Token` 和 `Encrypt Key`。
- `card.action.trigger` 只配置到 `/feishu/card-actions`。
- 不要同时配置旧版消息卡片请求网址或 `card.action.trigger_v1`。
- 回调保存成功不等于业务端到端成功；还要在测试群发真实任务验证卡片发送和点击。

飞书第一阶段权限至少包括：

- `im:message.group_at_msg:readonly`：接收群里 @机器人的消息。
- `im:message.p2p_msg:readonly`：接收发给机器人的单聊消息，建议一起开。
- `im:message:send_as_bot`：机器人发送任务确认卡、审批卡和进度卡。
- `im:chat:readonly`：获取机器人所在群列表和群信息，用于 `FEISHU_DEFAULT_CHAT_ID`。
- `card.action.trigger`：在“事件与回调 -> 回调配置”里配置到 `/feishu/card-actions`。

第二阶段 Bitable 同步再开 `bitable:app` 或后台等价的“查看、评论、编辑和管理多维表格 / 新增记录 / 批量新增记录 / 查询记录”权限，并把当前应用加入多维表格协作者可编辑范围。

## 12. 真实业务冒烟

按这个顺序试跑，不要跳过人工确认：

1. 邀请机器人进入测试群。
2. 在测试群 @机器人发一个简单职位任务。
3. 后端应快速 ACK，并在 `feishu_event_logs` 记录原始事件。
4. `feishu_outbox(kind=card_send)` 应出现任务确认卡发送任务。
5. 你在飞书卡片上 approve/reject。当前 edit 表单仍待实现，不能写成已通过。
6. approve 后才会写入 `graph_dispatch` outbox，由 worker 进入 graph。
7. 高风险副作用继续通过飞书确认卡走 HumanApproval。
8. 人工确认后才同步 Bitable。
9. 检查 Bitable 表是否出现预期记录，并核对 `feishu_bitable_write_chunks` 成功状态。

如果任何一步依赖真实飞书、真实 Bitable、真实模型或真实服务器网络，未实际跑通前都标记为“未验证”。

## 15. 故障排查

### Deploy Key 与仓库地址混淆

仓库 SSH 地址是：

```text
git@github.com:YiyuZh/headhunt-agent.git
```

Deploy Key 是服务器上 `.pub` 文件里的公钥内容：

```bash
cat ~/.ssh/id_ed25519_headhunt_agent.pub
```

多项目服务器上只配置 `Host github.com-headhunt-agent`，不要覆盖全局 `Host github.com`。

### `Could not resolve hostname github.com-headhunt-agent`

这说明还没有在服务器 `~/.ssh/config` 里配置项目级 SSH alias。`github.com-headhunt-agent` 不是公网域名，而是本机 SSH 配置里的别名。

先创建 key 和 SSH config：

```bash
mkdir -p ~/.ssh
chmod 700 ~/.ssh
test -f ~/.ssh/id_ed25519_headhunt_agent || ssh-keygen -t ed25519 -C "deploy-headhunt-agent" -f ~/.ssh/id_ed25519_headhunt_agent
touch ~/.ssh/config

grep -q '^Host github.com-headhunt-agent$' ~/.ssh/config || cat >> ~/.ssh/config <<'EOF'
Host github.com-headhunt-agent
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519_headhunt_agent
    IdentitiesOnly yes
EOF

chmod 600 ~/.ssh/config ~/.ssh/id_ed25519_headhunt_agent
cat ~/.ssh/id_ed25519_headhunt_agent.pub
```

把最后一行 `.pub` 公钥内容添加到 GitHub 仓库 `YiyuZh/headhunt-agent` 的 `Settings -> Deploy keys` 后，再验证：

```bash
ssh -T github.com-headhunt-agent
git ls-remote git@github.com-headhunt-agent:YiyuZh/headhunt-agent.git refs/heads/main
```

看到 `successfully authenticated, but GitHub does not provide shell access` 是成功。

### `ssh -T` 看起来像失败

如果输出包含：

```text
successfully authenticated, but GitHub does not provide shell access
```

这就是成功。GitHub 不提供 SSH shell，不影响 `git clone` / `git pull`。

### `POSTGRES_PASSWORD is missing`

说明 `.env` 没创建或没有被 Compose 读到。执行：

```bash
pwd
ls -la .env
bash scripts/lietou-oneclick.sh
docker compose config --quiet
```

必须在 `/opt/apps/headhunt-agent` 根目录运行启动命令。

### `status=not_ready`

看 `blocking_issues`，不要只看最后一行。缺第一阶段飞书值时，先补 `.env`，再运行：

```bash
python3 -m app.runtime.local_doctor --strict
```

如果输出类似 `feishu_required still placeholder`，说明 `.env` 里仍有飞书占位值。按 [飞书接入操作手册](飞书接入操作手册.md) 获取并填写：

- `FEISHU_APP_ID` / `FEISHU_APP_SECRET`：飞书开放平台应用凭证。
- `FEISHU_VERIFICATION_TOKEN` / `FEISHU_ENCRYPT_KEY`：事件订阅里的回调校验配置。
- `FEISHU_DEFAULT_CHAT_ID`：机器人所在测试群的 `chat_id`，通常以 `oc_` 开头。

如果只看到 `Optional Bitable business sync is not ready`，服务可以继续启动；等第二阶段要写多维表格时，再填写 `FEISHU_BITABLE_APP_TOKEN` 和 4 个 `FEISHU_BITABLE_*_TABLE_ID`。

### HTTPS 证书失败

检查：

```bash
dig +short headhunt.zenithy.art
curl -I http://headhunt.zenithy.art/health
cd /opt/apps/hiremate
docker compose logs caddy --tail=100
```

确认域名解析到服务器公网 IP，云安全组放行 `80/443`，并且公网端口只由 `hiremate-caddy` 监听。

### Caddy 502

当前服务器应使用统一网关。先看两个 compose 项目和共享网络：

```bash
cd /opt/apps/headhunt-agent
docker compose ps
docker network inspect shared_gateway
cd /opt/apps/hiremate
docker compose ps caddy
docker compose logs caddy --tail=100
```

统一网关的上游必须是 `lietou-api:8000`。如果 `hiremate-caddy` 没加入 `shared_gateway`，会解析不到 `lietou-api`；不要把上游盲目改成宿主机 `127.0.0.1`。

### Docker build 很慢

先区分是网络慢还是依赖冲突：

```bash
docker compose build --progress=plain api
```

如果错误类似：

```text
Get "https://registry-1.docker.io/v2/": net/http: request canceled
```

这是 Docker Hub 拉基础镜像失败，不是 Python pip 慢。先单独测试：

```bash
docker pull python:3.11-slim
```

如果这一步也超时，处理 Docker 拉镜像网络：

```bash
sudo mkdir -p /etc/docker
sudo nano /etc/docker/daemon.json
sudo systemctl daemon-reload
sudo systemctl restart docker
docker pull python:3.11-slim
```

`daemon.json` 里的 `registry-mirrors` 要使用你当前云厂商或企业网络可用的 Docker Hub 镜像加速地址。不要把 PyPI 镜像地址填进 Docker `registry-mirrors`，它们不是同一种服务。

如果 Docker 基础镜像已经能拉取，但卡在 `pip install`，再看本项目的 pip 镜像设置。本项目默认已经把 Docker build 阶段的 pip 源设为清华镜像：

```env
PIP_INDEX_URL=https://pypi.tuna.tsinghua.edu.cn/simple
PIP_DEFAULT_TIMEOUT=120
PIP_RETRIES=5
```

你仍然直接运行：

```bash
docker compose up -d --build
```

如果想确认 build 时确实使用了镜像源，用：

```bash
docker compose build --progress=plain api 2>&1 | grep -E 'Looking in indexes|pypi.tuna'
```

如果清华源临时异常，可以临时切回官方源：

```bash
PIP_INDEX_URL=https://pypi.org/simple docker compose build --progress=plain api
docker compose up -d
```

如果日志已经不是下载慢，而是 dependency conflict、build wheel failed、no matching distribution 之类错误，不要只升级单个包；先看 `pyproject.toml`、`Dockerfile` 和完整错误，按一组兼容版本调整。

### 飞书回调保存失败

先确认公网访问：

```bash
curl -f https://headhunt.zenithy.art/health
```

再确认 `.env` 里的 `FEISHU_VERIFICATION_TOKEN` 和 `FEISHU_ENCRYPT_KEY` 与飞书后台一致。
