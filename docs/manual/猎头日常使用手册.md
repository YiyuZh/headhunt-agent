# 猎头日常使用手册

## 状态

本文描述 Discord First 第一版完成后的目标日常操作流。当前代码已实现内部 `/tasks/authorize`、只读审计查询和内部审批 API，但 Discord slash commands、任务确认卡、button、modal 和 War Room thread/message 尚未实现；真实 Discord 联调未验证。

## 典型工作流

### 1. 发起任务

在允许的 Discord channel 输入：

```text
/headhunt new role_title:AI 平台后端负责人 location:上海 output_goal:生成岗位校准和人才地图
```

也可以筛候选人：

```text
/headhunt candidate role_title:AI 平台后端负责人 candidate_text:<候选人摘要>
```

系统会先快速 ephemeral ACK：

```text
已收到，正在解析任务。
```

这个 ACK 不代表 graph 已完成，只代表请求已验签、落库并进入后台队列。

### 2. Double Check 结构化任务

系统会把自然语言拆成 `CanonicalTaskBrief`，并发任务确认卡。

确认卡展示：

```text
任务类型
用户目标
岗位摘要
候选人摘要
输出目标
缺失字段
风险等级
推荐 council_mode
mode_reason
系统假设
```

检查确认卡时，优先看三件事：

```text
关键字段是否都有 field_source。
confidence 低的字段是否应该改成系统假设或让你补充。
没有来源的内容是否没有进入事实字段。
```

你可以：

```text
Approve：确认结构化任务无误，启动正式 graph。
Edit：打开 modal 修改字段，系统重新解析并再次确认。
Reject：拒绝任务，系统停止处理并记录原因。
```

Double check 是防止系统误解任务的关键步骤。未 approve 的任务不得进入正式 graph。Approve 后，本次 `CanonicalTaskBrief.version` 会被冻结；下游 Agent 只能读取冻结版 `CanonicalTaskBrief`、ArtifactRef、MemoryRef、SOPRef 和 SourceRef，不能各自重新理解原始任务或原始简历。

### 3. 查看进度

Approve 后，系统会创建 Discord War Room thread/message，并自动发送进度卡。

进度卡展示：

```text
当前阶段
council_mode
mode_reason
Agent 名称
读取的 artifact_refs
注入的 memory_refs
命中的 sop_refs
ReviewGate 结果
token 估算
下一步
```

进度卡不展示原始 prompt、不展示隐私全文、不展示 chain-of-thought。

ReviewGate 是每个关键 artifact 后面的质量门，不是全局聊天式审查。它只审查当前产物，并通过 conditional edge 路由：

```text
pass：进入下一节点。
needs_fix：只回到当前 artifact 对应的 repair_node 修一次。
needs_human：进入人工确认。
```

如果 repair 后第二次仍是 `needs_fix`，系统必须转为 `interrupt_human_approval`，不得重跑完整 graph 或回到无关上游节点。

### 4. 回答追问

如果信息不足，系统会发送追问卡。你可以在 Discord modal 里补充：

```text
薪酬范围
地点优先级
必须避开的公司
候选人硬性条件
交付时间
```

补充信息会形成新的结构化 artifact，并继续进入 graph。

### 5. 审批副作用

当系统准备执行高风险动作时，会发确认卡。

你可以：

```text
Approve：同意执行。
Reject：拒绝执行。
Edit：修改后执行，必须在 modal 中提交 edited_payload。
```

重复点击不会重复执行，因为系统使用 `idempotency_key`。

仍必须单独确认的动作：

```text
业务数据写入
报告发布
外部触达发送
候选人推荐结论保存
长期记忆进入 active
```

任务授权后的进度卡、追问卡、确认卡和结果卡可以自动发送。

### 6. 查看结果

结果卡应包含：

```text
本次结论摘要
council_mode
mode_reason
关键 evidence_refs / source_refs
生成的 artifact_refs
命中的 memory_refs
命中的 sop_refs
ReviewGate 结果
哪些动作已执行
哪些动作需要后续确认
```

你也可以查询：

```text
/headhunt status thread_id:<thread_id>
```

## 强制三省六部

当你希望更严格审查，直接写：

```text
请用三省六部完整会审
```

系统必须使用：

```text
council_mode: full_council
mode_reason: 用户明确要求三省六部
```

完整会审只是增加角色审查深度，不改变最小上下文原则。每个 Agent 仍只读取允许字段、artifact refs、memory refs、sop refs 和 source refs。

三省六部基于同一份已冻结 Canonical Context 做角色审查，不允许 Agent 之间自由聊天，也不允许某个 Agent 把自己的完整 prompt 或 ContextPack 转交给下一个 Agent。

## 你可以要求系统做什么

- 校准岗位需求。
- 拆解目标公司和人才池。
- 生成候选人筛选证据。
- 比较多个候选人。
- 生成报告草稿。
- 总结项目复盘。
- 提议长期记忆。
- 展示本次注入了哪些 memory_refs / sop_refs，以及为什么命中。

## 你不能让系统绕过什么

即使你授权了任务，以下动作仍必须单独确认：

- 写入业务数据库或外部系统。
- 发布报告。
- 创建外部任务。
- 发送外部触达。
- 提交候选人推荐结论。
- 把长期记忆置为 active。

## 推荐使用习惯

- 给岗位时先给硬条件和排除条件。
- 对候选人判断尽量提供证据来源。
- 需要全面审查时明确说“三省六部”。
- 如果系统解析错了，优先用 double check 的 Edit 修正。
- 对系统错误判断使用“纠正为...”的格式，方便进入 UserCorrectionMemory 审批。

## 事实、记忆和关键词边界

- 公司背调、当前在职、融资、新闻、裁员、组织变化等当前事实必须有 `SearchGateway source_refs`。
- 长期记忆不能替代实时搜索；它只用于 taxonomy、项目规则、用户修正、历史案例和 SOP 提示。
- 简历关键词抽取以当前简历 artifact 和当前 JD artifact 为主，长期记忆只能辅助解释行业词表和项目偏好。
- MemoryGateway 检索必须按 `thread_id / tenant_id / guild_id / user_id / project_id / requisition_id / candidate_id` 做 scope filter，命中的 memory_refs 应在卡片里说明命中原因和 token 估算。

## 当前可用的内部调试入口

当前未接真实 Discord 时，可用 HTTP 入口模拟“已确认授权”：

```bash
curl -X POST http://127.0.0.1:8000/tasks/authorize \
  -H "Content-Type: application/json" \
  -H "X-Internal-Admin-Key: <INTERNAL_ADMIN_API_KEY>" \
  -d '{
    "request_text": "请用三省六部完整会审 AI 平台负责人岗位并生成人才地图",
    "source": "manual",
    "source_ref": "manual-20260604-001",
    "approved": true,
    "approver": {"source": "manual", "user": "operator"}
  }'
```

查看线程和运行审计：

```bash
curl -H "X-Internal-Admin-Key: <INTERNAL_ADMIN_API_KEY>" \
  http://127.0.0.1:8000/threads/<thread_id>
curl -H "X-Internal-Admin-Key: <INTERNAL_ADMIN_API_KEY>" \
  http://127.0.0.1:8000/runs/<run_id>
```

内部审批：

```bash
curl -H "X-Internal-Admin-Key: <INTERNAL_ADMIN_API_KEY>" \
  http://127.0.0.1:8000/approvals/<interrupt_id>
```

```bash
curl -X POST http://127.0.0.1:8000/approvals/<interrupt_id>/decision \
  -H "Content-Type: application/json" \
  -H "X-Internal-Admin-Key: <INTERNAL_ADMIN_API_KEY>" \
  -d '{
    "decision": "approve",
    "approver": {"source": "internal", "user": "operator"}
  }'
```
