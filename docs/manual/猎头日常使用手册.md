# 猎头日常使用手册

本文描述 Feishu First 第一版目标使用方式。当前已有飞书事件/卡片基础代码、任务 intake 和确认卡排队基础，但真实飞书后台、真实卡片发送、飞书 War Room 和 graph 端到端联调仍未验证。

## 日常流程

以下是第一版目标流程，不代表当前已经完成真实飞书端到端联调。真实飞书后台、任务确认卡、War Room、Bitable 和 graph 联调完成前，以 `/ready`、`lietou-config-check --strict` 和本手册“未验证”边界为准。

```text
在飞书群 @机器人发起任务
-> 系统解析任务并发送结构化确认卡
-> 用户 approve / edit / reject
-> approve 后进入 AI 工作流
-> 飞书 War Room 展示进度、证据、ReviewGate 和待确认动作
-> 高风险动作通过飞书卡片人工确认
-> 批准后同步 Bitable 展示表
```

## 发起任务

在允许的飞书群里发送：

```text
@AI猎头机器人 新建岗位：北京 AI 产品经理，目标是生成岗位校准和人才地图
```

也可以发送候选人筛选任务：

```text
@AI猎头机器人 筛选候选人：这是简历摘要... 目标岗位是...
```

附件、JD、简历原文应作为 artifact 存储；War Room 默认只展示摘要和引用。

## 任务确认卡

当前代码会排队发送飞书任务确认卡；真实飞书卡片发送未联调前仍按“未验证”处理。确认卡展示：

```text
任务类型
岗位/候选人/输出目标
关键字段 field_source / confidence
缺失字段
系统 assumptions
推荐 council_mode / mode_reason
将注入的 ArtifactRef / MemoryRef / SOPRef / SourceRef 摘要
```

用户动作：

- `Approve`：确认理解无误，冻结 `CanonicalTaskBrief.version`，进入 graph。
- `Edit`：打开飞书卡片表单，修改后重新解析并再次确认。
- `Reject`：停止任务，原因可形成用户修正记忆提案。

未 approve 不得启动正式 graph。

## 进度与审批

目标实现后，飞书 War Room 会显示：

```text
council_mode 和 mode_reason
当前阶段
ReviewGate 结果
证据和 source_refs
memory_refs / sop_refs 命中原因
待人工确认动作
```

以下动作必须人工确认：

```text
业务表写入
Bitable 同步
候选人推荐结论保存
报告发布
外部触达
长期记忆 active
```

任务授权后的进度卡、追问卡、确认卡和结果卡可以自动发送，但不等于自动执行副作用。

## 模型配置

第一版目标通过飞书模型配置卡完成 BYOK：

```text
provider: openai / deepseek
usage: chat（当前飞书配置卡只开放 chat；embedding profile 仍需后续补齐飞书入口）
model_name
display_name
api_key
base_url 可选
```

API Key 加密保存，不在飞书、日志或 AgentRuns 中展示。自定义 `base_url` 只允许 HTTPS，服务端会拒绝 localhost/private IP，并在真实出站请求前再次做 DNS/private-IP 校验。

## 内部调试

当前未完成真实飞书端到端时，可用内部 API 模拟授权和审批，但不能把调试 API 当作普通用户入口：

```bash
curl -X POST https://<DOMAIN>/tasks/authorize \
  -H "X-Internal-Admin-Key: <key>" \
  -H "Content-Type: application/json" \
  -d '{"approved": true, "source_ref": "manual://debug"}'
```

正式路径必须回到：

```text
/feishu/events -> double check -> /feishu/card-actions -> graph/outbox
```
