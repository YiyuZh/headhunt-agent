# 飞书接入操作手册

> 状态更新：第一版主入口已经切换为 Discord First。本手册只保留给后续 Feishu/Bitable deferred adapter 或历史代码联调使用，不再作为第一版用户接入路径。第一版请优先使用 `docs/manual/Discord接入操作手册.md`。

## 状态

本文是人工接入飞书时使用的操作清单。当前已实现飞书事件回调 verifier、
卡片回调 verifier、事件/卡片入库投递 durable outbox、outbox dispatcher、
FeishuAuthProvider、卡片发送/更新 gateway、Bitable batch_create gateway、Bitable
chunk / record_id 映射、ArtifactStore、MemoryGateway 和 LangGraph 最小图骨架；
AgentHarness、OpenAI Responses LLMGateway、OpenAI EmbeddingGateway、War Room 卡片
outbox、ActionGate 提案路径、真实 LangGraph `interrupt()` / `Command(resume=...)`
恢复路径和审批后的 Bitable outbox 排队也已接入；
尚未创建真实飞书应用、尚未配置真实回调、尚未联调，所以真实后台步骤仍未验证。
当前飞书 outbox 的 `graph_dispatch` 只有在 runtime factory 注入真实 AgentHarness
后才会生产成功；`resume` 只有在 AgentHarness、ActionGate、ActionExecutor 都装配后
才会生产成功。真实 PostgreSQL checkpointer 和真实飞书点击恢复尚未联调。

如果后续重新启用 Feishu/Bitable adapter，也必须复用 Discord First 的核心业务门：
`TaskIntakeParser -> double check -> 冻结 CanonicalTaskBrief.version -> graph_dispatch`，
以及 artifact-level ReviewGate 的 `pass / needs_fix / needs_human` conditional edge。
飞书只能替换交互入口，不能绕过字段来源、人工确认、记忆 scope filter 或实时搜索边界。

## 第一步：创建飞书应用

1. 打开飞书开放平台。
2. 创建企业自建应用。
3. 进入应用详情页，记录：
   - `App ID`
   - `App Secret`
4. 进入“事件与回调”相关页面，记录：
   - `Verification Token`
   - `Encrypt Key`
5. 将这些值填入 `.env`。

## 第二步：配置机器人

1. 开启机器人能力。
2. 设置机器人名称和头像。
3. 发布应用版本。
4. 把机器人添加到测试群。
5. 记录测试群 `chat_id`，填入：

```bash
FEISHU_DEFAULT_CHAT_ID=
```

`chat_id` 获取方式：
- 优先通过系统后续提供的调试接口或日志读取机器人收到消息事件中的 chat ID。
- 也可以使用飞书 OpenAPI 查询会话，具体方式以飞书后台 API 调试台为准。

## 第三步：配置权限

按最小权限开通。具体 scope 名称以飞书后台当前页面为准。

建议权限范围：

| 能力 | 用途 |
| --- | --- |
| 发送消息 / 以应用身份发送消息 | War Room 卡片 |
| 接收消息事件 | 用户通过机器人发起任务 |
| 卡片回传交互 | approve / reject / edit |
| 多维表格读取 | 读取岗位、候选人或任务数据 |
| 多维表格写入 / 编辑 | 写入经人工确认后的业务记录 |
| 获取用户 open_id | 记录审批人 |

开通后需要发布应用版本，某些企业权限可能需要管理员审批。

## 第四步：配置回调 URL

本地调试需要 HTTPS 公网地址。

```text
事件订阅 URL: https://your-domain/feishu/events
卡片回调 URL: https://your-domain/feishu/card-actions
```

注意：
- 事件 URL 验证请求会带 challenge，服务需要快速原样返回。
- 事件订阅回调使用 Verification Token 校验；配置 Encrypt Key 时还会执行事件
  Encrypt Key 签名校验和解密。
- 卡片回调需要快速响应 toast 或卡片更新。
- 卡片回调安全校验使用 Verification Token 的 SHA-1 签名；飞书卡片回调不支持
  使用 Encrypt Key 做签名校验。
- graph 不在回调请求内同步执行。
- 事件订阅和卡片回调在飞书后台属于不同配置区域，不要把 `card.action.trigger` 混进事件订阅列表。

## 第五步：配置事件订阅

建议先开最小事件：

- `im.message.receive_v1`：接收消息事件，用于用户通过机器人发起任务。
- `im.chat.member.bot.added_v1`：机器人进群事件，用于记录机器人可用范围。
- 多维表格记录变更事件：可选，第一版可先手动导入；具体事件名以后台为准。

## 第六步：配置卡片回调

1. 进入应用详情页。
2. 打开“事件与回调”。
3. 进入“回调配置”页签。
4. 填写卡片回调 URL：

```text
https://your-domain/feishu/card-actions
```

5. 订阅新版卡片回传交互 `card.action.trigger`。
6. 不要同时启用旧版卡片回传，避免双回调。
7. 保存并发布应用版本。

## 第七步：配置多维表格

你需要准备一个飞书多维表格，并建立这些表：

| 表 | 用途 | 环境变量 |
| --- | --- | --- |
| 岗位需求 | JD、岗位状态、业务约束 | `FEISHU_BITABLE_REQUISITION_TABLE_ID` |
| 候选人 | 候选人摘要、公司、title、状态 | `FEISHU_BITABLE_CANDIDATE_TABLE_ID` |
| 人才地图 | 目标公司、候选人线索、证据 | `FEISHU_BITABLE_TALENT_MAP_TABLE_ID` |
| 报告 | 报告草稿、审批状态、发布状态 | `FEISHU_BITABLE_REPORT_TABLE_ID` |

记录多维表格 app token：

```bash
FEISHU_BITABLE_APP_TOKEN=
```

取值位置：
- `app_token`：从多维表格 URL 或飞书 API 调试台获取。
- `table_id`：从具体数据表 URL 或“列出数据表”接口获取。
- 当前可用 `/ready`、`python -m app.runtime.config_check` 或
  `lietou-config-check --strict` 检查 app_token/table_id 等配置是否已填写。
  注意：配置检查只确认环境变量齐全，不等于已经验证飞书权限或表格真实可访问。
- 真实飞书配置完成后，可运行 `python -m app.runtime.external_smoke --skip-openai`
  或 `lietou-external-smoke --skip-openai` 验证 tenant token、默认群信息和 Bitable
  `records/search` 读取权限；该命令不发送卡片、不写记录。

## 第八步：测试动作

1. 在飞书群 @ 机器人：

```text
帮我会审这个岗位：AI 平台后端负责人，需要懂 LLM 应用、平台工程和团队管理。
```

2. 当前阶段先检查 `/feishu/events` 是否 ACK 并生成 outbox。
3. 启动 durable outbox worker：

```bash
lietou-outbox-worker
```

4. 注入真实 AgentHarness runtime 后，检查 War Room 进度卡和 AgentRun 卡片。
5. War Room 进度卡应展示：
   - `council_mode`
   - `mode_reason`
   - memory_refs
   - artifact_refs
6. 先检查任务 double check 卡：关键字段必须有 `field_source` 和 `confidence`；无来源推断只能进入 `assumptions`；Approve 后冻结 `CanonicalTaskBrief.version`。
7. 检查 ReviewGate：它只审查当前 artifact，`needs_fix` 只回对应 `repair_node` 一次，第二次仍失败必须人工确认。
8. 对确认卡执行 approve / reject / edit。
9. 检查 Bitable 是否只在人工确认后写入。
   - approve / edit：生成 `HumanApproval`，恢复同一个 `thread_id`，排队
     `bitable_write` outbox。
   - reject：只记录拒绝，不写业务表。
   - 缺少或不匹配 `thread_id/action_id/interrupt_id/idempotency_key` 的卡片回调会被拒绝。
10. 检查公司当前事实必须来自 `SearchGateway source_refs`，简历关键词抽取以当前简历/JD artifact 为主；长期记忆只提供 taxonomy、项目规则、用户修正、历史案例和 SOP。

## 你需要给 Codex / 系统的数据

| 数据 | 用途 | 放在哪里 |
| --- | --- | --- |
| 飞书 App ID | 调用 OpenAPI | `.env FEISHU_APP_ID` |
| 飞书 App Secret | 获取 tenant token | `.env FEISHU_APP_SECRET` |
| Verification Token | 回调校验 | `.env FEISHU_VERIFICATION_TOKEN` |
| Encrypt Key | 加密回调验签/解密 | `.env FEISHU_ENCRYPT_KEY` |
| 测试群 chat_id | War Room 默认群 | `.env FEISHU_DEFAULT_CHAT_ID` |
| Bitable app_token | 多维表格应用 | `.env FEISHU_BITABLE_APP_TOKEN` |
| 各业务表 table_id | 写入岗位/候选人/人才地图/报告 | `.env FEISHU_BITABLE_*_TABLE_ID` |
| LLM / embedding API Key | 模型调用和向量化 | `.env LLM_API_KEY` |
| 一条测试 JD | 首次联调 | API 或飞书消息 |
| 几条测试候选人或历史案例 | 检验筛选和记忆 | API 或多维表格 |

## 不要给系统的数据

- 不要把真实候选人联系方式放在未授权的测试数据里。
- 不要把生产 App Secret 写进文档或提交到 git。
- 不要让测试机器人进入真实业务群，直到完成联调验收。
