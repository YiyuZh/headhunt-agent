# 飞书接入操作手册

本文是第一版飞书接入手册。目标是把飞书机器人、事件订阅、交互卡片、War Room 群消息和 Bitable 同步接到 `<repo-root>` 后端。服务器推荐路径为 `/opt/apps/headhunt-agent`，正式域名为 `headhunt.zenithy.art`；Windows 本机 clone 为 `D:\apps\headhunt-agent`。真实飞书后台、真实 Bitable 权限、真实卡片发送和 graph 端到端联调未完成前，必须写“未验证”。

## 你需要准备

- 一个飞书开放平台应用。
- 一个测试群，机器人已被邀请进群。
- 一个可公网访问的 HTTPS 域名，正式部署使用 `headhunt.zenithy.art`。
- 服务器已能执行 `bash scripts/lietou-oneclick.sh --start`。
- OpenAI / DeepSeek API Key 由用户后续通过飞书模型配置卡录入，不写入共享 `.env`。

Deploy Key、DNS、Caddy、备份和回滚按 [服务器部署与使用手册](本地部署与使用手册.md) 执行；服务器 Deploy Key 使用 `github.com-headhunt-agent` alias 和 `~/.ssh/id_ed25519_headhunt_agent`。本文只写飞书后台和业务联调。

## 创建飞书应用

1. 打开 [飞书开放平台](https://open.feishu.cn/)。
2. 创建企业自建应用。
3. 记录 `App ID`、`App Secret`。
4. 在“事件订阅”里记录或配置 `Verification Token` 和 `Encrypt Key`。
5. 开启机器人能力，并把机器人加入测试群。

## 配置回调地址

服务器正式地址为：

```text
https://headhunt.zenithy.art
```

在飞书后台配置：

```text
事件订阅 URL: https://headhunt.zenithy.art/feishu/events
卡片回调 URL: https://headhunt.zenithy.art/feishu/card-actions
```

要求：

- `/feishu/events` 必须能处理 URL verification / challenge。
- `/feishu/events` 业务事件只做验签、幂等落库、outbox 和 ACK。
- `/feishu/card-actions` 只做验签、卡片动作解析、HumanApproval 或 double check 入库、outbox 和 toast。
- `card.action.trigger` 使用新版卡片回传交互回调，统一配置到 `/feishu/card-actions`。
- 两个回调入口都使用同一组 `Verification Token` 和 `Encrypt Key`；配置 Encrypt Key 后，服务端会先按 raw body 校验签名，再解密 `encrypt` payload，最后校验 token。
- 不要同时配置旧版消息卡片请求网址或旧版 `card.action.trigger_v1`，否则可能收到重复回调；如果误把 `card.action.trigger` 配到 `/feishu/events`，后端会返回 400 提醒改到 `/feishu/card-actions`。
- 不在回调请求里同步跑完整 graph、LLM、Bitable 写入或外部触达。

## 配置权限

第一版建议申请：

```text
机器人收发消息
读取群信息
发送和更新交互卡片
读取用户基础身份标识
Bitable app/table record 读写
```

权限以飞书后台实际名称为准。没有真实租户权限前，文档和测试结果只能标“未验证”。

## 配置 Bitable

1. 创建一个多维表格。
2. 为岗位、候选人、任务结果、审批结果建立展示表。
3. 记录 `app_token` 和 `table_id`。
4. 给应用授予对应 Bitable 权限。
5. 后端 `.env` 填入：

```env
FEISHU_BITABLE_APP_TOKEN=
FEISHU_BITABLE_REQUISITION_TABLE_ID=
FEISHU_BITABLE_CANDIDATE_TABLE_ID=
FEISHU_BITABLE_TALENT_MAP_TABLE_ID=
FEISHU_BITABLE_REPORT_TABLE_ID=
```

Bitable 只做业务展示和同步表。主数据、审批、outbox、AgentRuns、Artifacts、Memory 仍在 PostgreSQL。

## 配置 `.env`

先生成 `.env`：

```bash
bash scripts/lietou-oneclick.sh
```

然后至少填写飞书和 Bitable 真实值；本地运行密钥会由上面的命令自动生成：

```env
DOMAIN=headhunt.zenithy.art
INTERNAL_ADMIN_API_KEY=<auto-generated>
MODEL_SECRET_ENCRYPTION_KEY=<auto-generated>

POSTGRES_DB=lietou
POSTGRES_USER=lietou
POSTGRES_PASSWORD=<auto-generated>

FEISHU_BASE_URL=https://open.feishu.cn
FEISHU_APP_ID=
FEISHU_APP_SECRET=
FEISHU_VERIFICATION_TOKEN=
FEISHU_ENCRYPT_KEY=
FEISHU_DEFAULT_CHAT_ID=
FEISHU_BITABLE_APP_TOKEN=
FEISHU_BITABLE_REQUISITION_TABLE_ID=
FEISHU_BITABLE_CANDIDATE_TABLE_ID=
FEISHU_BITABLE_TALENT_MAP_TABLE_ID=
FEISHU_BITABLE_REPORT_TABLE_ID=
```

不要把用户的 OpenAI / DeepSeek Key 写进 `.env` 作为共享业务模型。第一版目标是用户通过飞书卡片 BYOK。

启动前执行：

```bash
python3 -m app.runtime.local_doctor --strict
```

该命令会检查 `.env`、飞书/Bitable 必填项、占位符、Docker 和 Docker Compose v2，不输出密钥值。

## 启动服务

在项目根目录执行：

```bash
bash scripts/lietou-oneclick.sh --start
```

```bash
curl https://headhunt.zenithy.art/health
curl -H "X-Internal-Admin-Key: <INTERNAL_ADMIN_API_KEY>" https://headhunt.zenithy.art/ready
```

本机没有 Docker 时，不能写成已通过；只记录“未验证”并在服务器执行。

## 联调顺序

1. 先保存 `/feishu/events`，确认 challenge 通过。
2. 再保存 `/feishu/card-actions`，确认签名校验通过。
3. 在测试群 @机器人发送一个简单任务。
4. 后端应 ACK，并把原始事件写入 `feishu_event_logs`，把飞书确认卡写入 `feishu_outbox(kind=card_send)`。
5. 当前已实现基础 TaskIntakeParser 和任务确认卡生成；真实飞书卡片发送未联调前仍标注“未验证”。
6. 用户 approve/reject；edit 表单仍待实现，不能写成已通过。
7. approve 后才会写入 `graph_dispatch` outbox；高风险动作继续通过飞书确认卡走 HumanApproval。
8. 人工确认后才同步 Bitable。

## 模型配置

第一版目标是通过飞书模型配置卡：

```text
provider: openai / deepseek
usage: chat（当前飞书配置卡只开放 chat；embedding profile 仍需后续补齐飞书入口）
model_name
display_name
api_key
base_url 可选
```

API Key 由 `MODEL_SECRET_ENCRYPTION_KEY` 加密保存，不在飞书卡片、日志、AgentRuns 或查询接口展示。

当前飞书内 BYOK 配置卡已有本地实现：若用户没有默认 chat 模型 profile，系统会发送“请先配置模型”表单卡；API Key 输入框按 password 类型构造，OpenAI/DeepSeek 提交按钮有独立 `name` 和 callback value；用户提交后，服务端会校验操作者与原任务 owner 的同一 ID namespace，拒绝 localhost/private IP 等不安全 `base_url`，并在真实 LLM/Embedding 出站请求前再次做 DNS/private-IP 拦截；通过后加密保存为默认 profile，更新原 setup 卡为“模型已保存”，并继续发送原任务确认卡。真实飞书卡片提交、卡片更新和真实模型 smoke 仍需在飞书后台联调验证。

## 未验证边界

- 真实飞书后台保存回调未验证。
- 真实飞书卡片发送/更新未验证。
- 真实 Bitable 写入未验证。
- 真实 OpenAI/DeepSeek 调用未验证。
- 真实 graph 端到端联调未验证。
