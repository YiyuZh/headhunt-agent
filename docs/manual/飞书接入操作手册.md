# 飞书接入操作手册

本文是第一版飞书接入手册。目标是把飞书机器人、事件订阅、交互卡片、War Room 群消息和 Bitable 同步接到 `<repo-root>` 后端。服务器推荐路径为 `/opt/apps/headhunt-agent`，正式域名为 `headhunt.zenithy.art`；Windows 本机 clone 为 `D:\apps\headhunt-agent`。真实飞书后台、真实 Bitable 权限、真实卡片发送和 graph 端到端联调未完成前，必须写“未验证”。

## 你需要准备

- 一个飞书开放平台应用。
- 一个测试群，机器人已被邀请进群。
- 一个可公网访问的 HTTPS 域名，正式部署目标使用 `headhunt.zenithy.art`。
- 服务器已有 `/opt/apps/hiremate` 统一 Caddy 网关项目；当前只确认 `ttcintern` 已配置 HTTPS，HeadHunt 站点块仍需按主部署手册配置和验证。`headhunt-agent` 可在 `/opt/apps/headhunt-agent` 执行 `docker compose up -d --build`。
- OpenAI / DeepSeek API Key 由用户后续通过飞书模型配置卡录入，不写入共享 `.env`。

当前实际状态要分清：已知只有 `ttcintern` 站点完成 HTTPS 配置；`headhunt.zenithy.art` 必须等统一 Caddy 新增站点块、reload 成功并通过 `curl -f https://headhunt.zenithy.art/health` 后，才算可用于飞书回调。

Deploy Key、DNS、统一 Caddy 网关、备份和回滚按 [服务器部署与使用手册](本地部署与使用手册.md) 执行；服务器 Deploy Key 使用 `github.com-headhunt-agent` alias 和 `~/.ssh/id_ed25519_headhunt_agent`。本文只写飞书后台和业务联调。

## 创建飞书应用

1. 打开 [飞书开放平台](https://open.feishu.cn/)。
2. 创建企业自建应用。
3. 左侧进入“凭证与基础信息”，记录 `App ID` 和 `App Secret`。
4. 左侧进入“添加应用能力”，开启“机器人”能力。
5. 在飞书客户端里创建一个测试群，把这个机器人邀请进群。

`App ID` 通常长得像 `cli_xxxxxxxxxxxxxxxx`。`App Secret` 是密钥，只能填到服务器 `/opt/apps/headhunt-agent/.env`，不要写进文档、README、截图、聊天记录或 GitHub。

## 先按飞书后台配置事件与回调

你截图里的位置是“开发配置 -> 事件与回调”。当前页面有三个关键页签：

- “事件配置”：配置普通事件，例如机器人收到群消息。
- “回调配置”：配置卡片按钮、表单提交等交互回调。
- “加密策略”：配置 `Verification Token` 和 `Encrypt Key`。

本项目目标是使用公网 HTTPS 开发者服务器回调，所以不要选“使用长连接接收事件/回调”。应选择“将事件发送至开发者服务器”，让飞书直接 POST 到服务器。只有 `https://headhunt.zenithy.art/health` 已经能访问时，才去飞书后台验证和保存下面两个回调地址。

### 1. 配置事件订阅 `/feishu/events`

1. 在飞书开放平台进入当前应用。
2. 左侧点击“事件与回调”。
3. 打开“事件配置”页签。
4. 在“订阅方式”右侧点击铅笔图标。
5. 选择“将事件发送至开发者服务器”。
6. 请求地址填写：

```text
https://headhunt.zenithy.art/feishu/events
```

7. 点击“验证”或“保存”。飞书会发送 challenge；后端通过后，页面会显示验证成功。
8. 在“已添加事件”区域点击“添加事件”，搜索并添加：

```text
接收消息 v2.0 / im.message.receive_v1
```

`im.message.receive_v1` 是机器人收到消息后触发的事件。本项目靠它接收你在测试群里 @机器人的任务。

### 2. 配置卡片回调 `/feishu/card-actions`

1. 仍在“事件与回调”页面，切到“回调配置”页签。
2. 请求地址填写：

```text
https://headhunt.zenithy.art/feishu/card-actions
```

3. 添加回调：

```text
卡片回传交互 / card.action.trigger
```

4. 保存配置。

只配置新版 `card.action.trigger`。不要配置旧版 `card.action.trigger_v1`，也不要把 `card.action.trigger` 配到 `/feishu/events`，否则卡片按钮点击可能无法被当前后端正确处理。

### 3. 配置加密策略里的两个值

切到“加密策略”页签，找到并记录：

- `Verification Token`
- `Encrypt Key`

这两个值的意思是：

- `Verification Token`：飞书在 challenge、事件、卡片回调里带上的校验 token。后端会检查它是否等于 `.env` 里的 `FEISHU_VERIFICATION_TOKEN`。
- `Encrypt Key`：飞书加密回调内容时使用的密钥。后端会用 `.env` 里的 `FEISHU_ENCRYPT_KEY` 解密 `encrypt` payload。

它们不是本项目 `bash scripts/lietou-oneclick.sh` 生成的 `<auto-generated>`。必须从飞书后台复制，和服务器 `.env` 完全一致。事件订阅和卡片回调使用同一个应用的同一组 `Verification Token` / `Encrypt Key`。

如果你在飞书后台重新生成或修改了这两个值，需要同步改服务器 `.env`，然后重启 API：

```bash
cd /opt/apps/headhunt-agent
nano .env
docker compose up -d --build
```

### 4. 权限开通后要发布版本

你截图里顶部提示“应用发布后，当前配置方可生效”。这表示：权限、事件、机器人能力等配置保存后，还需要进入“版本管理与发布”，创建版本并发布到当前企业，真实用户侧才会生效。

测试阶段建议顺序是：

1. 先配置权限。
2. 再配置事件订阅、卡片回调、加密策略。
3. 创建版本并发布。
4. 把机器人加入测试群。
5. 回到服务器填 `.env`。
6. 再运行 `python3 -m app.runtime.local_doctor --strict`。

## 获取 `.env` 里的飞书 ID

`python3 -m app.runtime.local_doctor --strict` 里列出的 `FEISHU_*` 字段都要填真实值，不能保留空值、`change-this-*` 或示例占位。

| `.env` 字段 | 从哪里获取 | 识别方式 |
| --- | --- | --- |
| `FEISHU_APP_ID` | 飞书开放平台 -> 当前应用 -> 凭证与基础信息 -> App ID | 通常以 `cli_` 开头 |
| `FEISHU_APP_SECRET` | 飞书开放平台 -> 当前应用 -> 凭证与基础信息 -> App Secret | 只保存在服务器 `.env`，不要发到聊天或提交 GitHub |
| `FEISHU_VERIFICATION_TOKEN` | 飞书开放平台 -> 当前应用 -> 事件订阅 -> Verification Token | 事件回调和卡片回调使用同一组 token |
| `FEISHU_ENCRYPT_KEY` | 飞书开放平台 -> 当前应用 -> 事件订阅 -> Encrypt Key | 启用加密回调后复制，不是 `MODEL_SECRET_ENCRYPTION_KEY` |
| `FEISHU_DEFAULT_CHAT_ID` | 机器人已加入的飞书测试群 `chat_id` | 通常以 `oc_` 开头，不是群名称 |
| `FEISHU_BITABLE_APP_TOKEN` | 多维表格 URL 里 `/base/` 后面的 app token | 通常以 `base` 或 `bascn` 开头 |
| `FEISHU_BITABLE_*_TABLE_ID` | 多维表格 URL 的 `table=` 参数，或表格开发者信息里的 table ID | 通常以 `tbl` 开头，不是 `view`、`record` 或表名 |

### 获取 `FEISHU_DEFAULT_CHAT_ID`

`FEISHU_DEFAULT_CHAT_ID` 是默认 War Room 群 ID，不是群名称。它通常以 `oc_` 开头。系统需要它来知道默认把任务确认卡、审批卡和进度卡发到哪个飞书群。

推荐方式是用飞书 OpenAPI 拉取机器人所在群列表。先确认：

- 机器人已经加入目标测试群。
- “权限管理”里已经开通读取群信息相关权限。
- 应用版本已经发布，权限已经生效。
- `.env` 里已经填好 `FEISHU_APP_ID` 和 `FEISHU_APP_SECRET`。

然后在服务器执行：

```bash
cd /opt/apps/headhunt-agent
set -a
. ./.env
set +a

TENANT_ACCESS_TOKEN="$(
  curl -sS -X POST "https://open.feishu.cn/open-apis/auth/v3/tenant_access_token/internal" \
    -H "Content-Type: application/json" \
    -d "{\"app_id\":\"$FEISHU_APP_ID\",\"app_secret\":\"$FEISHU_APP_SECRET\"}" \
  | python3 -c "import json,sys; d=json.load(sys.stdin); assert d.get('code') == 0, d; print(d['tenant_access_token'])"
)"

curl -sS "https://open.feishu.cn/open-apis/im/v1/chats?page_size=50" \
  -H "Authorization: Bearer $TENANT_ACCESS_TOKEN" \
  | python3 -m json.tool
```

这段命令做了两件事：

- 先用 `FEISHU_APP_ID` / `FEISHU_APP_SECRET` 换取 `tenant_access_token`。
- 再用这个 token 读取机器人可见的群列表。

输出里找到目标测试群的 `name`，复制同一项里的 `chat_id` 到 `.env`：

```env
FEISHU_DEFAULT_CHAT_ID=oc_xxxxxxxxx
```

如果输出为空或报 `access denied`，优先检查机器人是否在群里、群信息权限是否开通、权限是否已发布。

### 获取 Bitable app token 和 table ID

打开目标多维表格，浏览器地址通常类似：

```text
https://example.feishu.cn/base/bascnxxxxxxxx?table=tblxxxxxxxx&view=vewxxxxxxxx
```

其中：

- `/base/` 后面的 `bascnxxxxxxxx` 是 `FEISHU_BITABLE_APP_TOKEN`。
- `table=` 后面的 `tblxxxxxxxx` 是当前表的 `table_id`。
- `view=` 后面的 `vewxxxxxxxx` 是视图 ID，不要填进 `.env`。

对应填写：

```env
FEISHU_BITABLE_APP_TOKEN=bascnxxxxxxxx
FEISHU_BITABLE_REQUISITION_TABLE_ID=tblxxxxxxxx
FEISHU_BITABLE_CANDIDATE_TABLE_ID=tblxxxxxxxx
FEISHU_BITABLE_TALENT_MAP_TABLE_ID=tblxxxxxxxx
FEISHU_BITABLE_REPORT_TABLE_ID=tblxxxxxxxx
```

第一版需要 4 个表 ID：

- `FEISHU_BITABLE_REQUISITION_TABLE_ID`：岗位/需求校准展示表。
- `FEISHU_BITABLE_CANDIDATE_TABLE_ID`：候选人判断展示表。
- `FEISHU_BITABLE_TALENT_MAP_TABLE_ID`：人才地图展示表。
- `FEISHU_BITABLE_REPORT_TABLE_ID`：报告/复盘展示表。

如果只是先让服务启动、暂时不验证 Bitable 写入，可以先用同一个测试多维表格里的真实 `app_token` 和 4 个真实测试表 `table_id`。这只能让 strict preflight 通过；真实业务写入仍必须在后续冒烟里单独验证。

## 配置回调地址

服务器正式地址为：

```text
https://headhunt.zenithy.art
```

在飞书后台配置：

```text
事件订阅 URL: https://headhunt.zenithy.art/feishu/events
卡片回调 URL: https://headhunt.zenithy.art/feishu/card-actions
```

要求：

- `/feishu/events` 必须能处理 URL verification / challenge。
- `/feishu/events` 业务事件只做验签、幂等落库、outbox 和 ACK。
- `/feishu/card-actions` 只做验签、卡片动作解析、HumanApproval 或 double check 入库、outbox 和 toast。
- `card.action.trigger` 使用新版卡片回传交互回调，统一配置到 `/feishu/card-actions`。
- 两个回调入口都使用同一组 `Verification Token` 和 `Encrypt Key`；配置 Encrypt Key 后，服务端会先按 raw body 校验签名，再解密 `encrypt` payload，最后校验 token。
- 不要同时配置旧版消息卡片请求网址或旧版 `card.action.trigger_v1`，否则可能收到重复回调；如果误把 `card.action.trigger` 配到 `/feishu/events`，后端会返回 400 提醒改到 `/feishu/card-actions`。
- 不在回调请求里同步跑完整 graph、LLM、Bitable 写入或外部触达。

## 配置权限

在飞书开放平台左侧点击“权限管理”，再点“开通权限”。你截图里的权限列表为空，所以这里必须先补齐。

第一版建议至少开通这些能力。后台名称可能会随飞书版本调整，按“权限管理”搜索框里最接近的权限选择：

```text
机器人接收群消息 / 接收 @机器人消息
机器人发送消息
读取群信息 / 获取群组信息
读取用户基础身份标识
多维表格 app/table/record 读取
多维表格 record 写入或管理
交互卡片发送、更新或回调相关权限
```

用途对应关系：

- `im.message.receive_v1` 事件需要机器人能接收群消息。
- 获取 `FEISHU_DEFAULT_CHAT_ID` 的群列表命令需要读取群信息权限。
- War Room 卡片发送和更新需要机器人发消息、交互卡片相关权限。
- Bitable 同步需要多维表格 app/table/record 读写权限。

权限保存后不要立刻跑服务器 strict。先进入“版本管理与发布”，创建版本并发布；如果企业要求管理员审核，等审核通过后再回服务器执行检查。没有真实租户权限前，文档和测试结果只能标“未验证”。

## 配置 Bitable

1. 创建一个多维表格。
2. 为岗位、候选人、任务结果、审批结果建立展示表。
3. 记录 `app_token` 和 `table_id`。
4. 在飞书开放平台“权限管理”里开通多维表格读写权限。
5. 如果多维表格本身有独立协作者/应用权限设置，把当前应用加入可访问范围，并授予可编辑权限。
6. 后端 `.env` 填入：

```env
FEISHU_BITABLE_APP_TOKEN=
FEISHU_BITABLE_REQUISITION_TABLE_ID=
FEISHU_BITABLE_CANDIDATE_TABLE_ID=
FEISHU_BITABLE_TALENT_MAP_TABLE_ID=
FEISHU_BITABLE_REPORT_TABLE_ID=
```

Bitable 只做业务展示和同步表。主数据、审批、outbox、AgentRuns、Artifacts、Memory 仍在 PostgreSQL。

## 联调顺序

1. 先保存 `/feishu/events`，确认 challenge 通过。
2. 再保存 `/feishu/card-actions`，确认签名校验通过。
3. 在测试群 @机器人发送一个简单任务。
4. 后端应 ACK，并把原始事件写入 `feishu_event_logs`，把飞书确认卡写入 `feishu_outbox(kind=card_send)`。
5. 当前已实现基础 TaskIntakeParser 和任务确认卡生成；真实飞书卡片发送未联调前仍标注“未验证”。
6. 用户 approve/reject；edit 表单仍待实现，不能写成已通过。
7. approve 后才会写入 `graph_dispatch` outbox；高风险动作继续通过飞书确认卡走 HumanApproval。
8. 人工确认后才同步 Bitable。

## 模型配置

第一版目标是通过飞书模型配置卡：

```text
provider: openai / deepseek
usage: chat（当前飞书配置卡只开放 chat；embedding profile 仍需后续补齐飞书入口）
model_name
display_name
api_key
base_url 可选
```

API Key 由 `MODEL_SECRET_ENCRYPTION_KEY` 加密保存，不在飞书卡片、日志、AgentRuns 或查询接口展示。

当前飞书内 BYOK 配置卡已有本地实现：若用户没有默认 chat 模型 profile，系统会发送“请先配置模型”表单卡；API Key 输入框按 password 类型构造，OpenAI/DeepSeek 提交按钮有独立 `name` 和 callback value；用户提交后，服务端会校验操作者与原任务 owner 的同一 ID namespace，拒绝 localhost/private IP 等不安全 `base_url`，并在真实 LLM/Embedding 出站请求前再次做 DNS/private-IP 拦截；通过后加密保存为默认 profile，更新原 setup 卡为“模型已保存”，并继续发送原任务确认卡。真实飞书卡片提交、卡片更新和真实模型 smoke 仍需在飞书后台联调验证。

## 未验证边界

- 真实飞书后台保存回调未验证。
- 真实飞书卡片发送/更新未验证。
- 真实 Bitable 写入未验证。
- 真实 OpenAI/DeepSeek 调用未验证。
- 真实 graph 端到端联调未验证。
