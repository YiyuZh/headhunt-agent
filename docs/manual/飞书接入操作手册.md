# 飞书接入操作手册

本文只处理飞书开放平台接入。服务器部署、Deploy Key、DNS、统一 Caddy 网关、备份和回滚按 [服务器部署与使用手册](本地部署与使用手册.md) 执行。

当前要分清两个阶段：

- 第一阶段：让机器人、事件订阅、交互卡片回调、War Room 群消息先跑通。不需要多维表格。
- 第二阶段：启用审批后的业务展示/同步，再配置 Bitable 多维表格。

`python3 -m app.runtime.local_doctor --strict` 现在只把第一阶段飞书必填项作为阻断项。`FEISHU_BITABLE_*` 缺失只会给出 Bitable 同步未就绪警告，不阻断服务启动和飞书回调。

## 你需要准备

- 一个飞书开放平台企业自建应用。
- 一个测试群，机器人已被邀请进群。
- 目标域名 `headhunt.zenithy.art`，并且先通过 `curl -f https://headhunt.zenithy.art/health`。
- 服务器路径 `/opt/apps/headhunt-agent`。
- 当前实际 HTTPS 状态要分清：已知只有 `ttcintern` 站点完成 HTTPS；HeadHunt 必须等统一 Caddy 新增站点块、reload 成功、DNS 生效、`/health` 通过后，才能去飞书后台保存回调。

不要把 App Secret、Verification Token、Encrypt Key、OpenAI Key 写进文档、截图、聊天或 GitHub。

## 一次性权限清单

在飞书开放平台进入当前应用，左侧点“权限管理”，点“开通权限”。建议优先用“权限标识”搜索；如果后台只显示中文名称，就按“后台搜索词”搜索。

### 第一阶段必须开通

| 权限标识/配置项 | 后台搜索词 | 用途 |
| --- | --- | --- |
| 机器人能力 | 添加应用能力 -> 机器人 | 让应用成为可进群的机器人 |
| `im:message.group_at_msg:readonly` | 获取用户在群组中 @ 机器人的消息、接收 @ 机器人消息 | 触发 `im.message.receive_v1`，用于群里 @bot 创建任务 |
| `im:message.p2p_msg:readonly` | 获取用户发给机器人的单聊消息、接收单聊消息 | 后续支持私聊机器人；建议一起开 |
| `im:message:send_as_bot` | 以应用的身份发消息、机器人发送消息 | 发送任务确认卡、审批卡、进度卡 |
| `im:chat:readonly` | 获取群组信息、读取群信息、获取用户或机器人所在群列表 | 获取 `FEISHU_DEFAULT_CHAT_ID`，检查机器人是否在测试群 |
| `card.action.trigger` | 卡片回传交互、卡片按钮回调 | 不是普通权限；在“事件与回调 -> 回调配置”里配置到 `/feishu/card-actions` |

如果飞书后台出现更细的“更新应用发送的消息 / 更新卡片 / 更新机器人消息”权限，也一起开。项目 worker 已有 `card_update` 路径，后续卡片状态刷新会用到。

### 第二阶段 Bitable 同步需要开通

| 权限标识/配置项 | 后台搜索词 | 用途 |
| --- | --- | --- |
| `bitable:app` | 查看、评论、编辑和管理多维表格；多维表格读写；多维表格记录管理 | 审批通过后写入岗位、候选人、人才地图、报告表 |
| 多维表格协作者权限 | 多维表格右上角分享/协作者/高级权限 | 把当前应用加入可访问范围，并授予可编辑权限 |

如果后台把 Bitable 权限拆成只读和写入，必须选择能“新增记录 / 批量新增记录 / 查询记录”的组合。只读权限只能用于 smoke 读取，不能完成业务写入。

### 以后扩展建议一次性申请

| 权限标识/配置项 | 后台搜索词 | 用途 |
| --- | --- | --- |
| `contact:user.id:readonly` | 获取用户 user ID、读取用户 ID | 统一识别点击卡片的人、任务 owner、审批人 |
| `contact:user.base:readonly` | 获取用户基本信息、读取用户基础信息 | 展示发起人/审批人名称，做审计报表 |
| `contact:department.base:readonly` | 获取部门基础信息、读取部门信息 | 后续按团队/部门做权限隔离和报表 |
| 飞书审批相关权限 | 审批定义、审批实例、审批事件 | 当前项目先用自研 ActionProposal/HumanApproval；只有以后接飞书 OA 审批流时再启用 |

不要开与当前项目无关的全量通讯录管理、企业管理、安全审计、日历、云文档全盘管理等高风险权限。权限保存后必须进入“版本管理与发布”，创建版本并发布；如果企业要求管理员审核，等审核通过后才会生效。

## 创建飞书应用

1. 打开 [飞书开放平台](https://open.feishu.cn/)。
2. 创建企业自建应用。
3. 左侧进入“凭证与基础信息”，记录 `App ID` 和 `App Secret`。
4. 左侧进入“添加应用能力”，开启“机器人”。
5. 在飞书客户端创建测试群，把机器人邀请进群。

`App ID` 通常以 `cli_` 开头。`App Secret` 是密钥，只能填到服务器 `/opt/apps/headhunt-agent/.env`。

## 配置事件与回调

你截图所在位置是“开发配置 -> 事件与回调”，页面通常有“事件配置 / 回调配置 / 加密策略”三个页签。

本项目使用公网 HTTPS 开发者服务器回调，不使用“长连接接收事件/回调”。只有 `https://headhunt.zenithy.art/health` 已能访问时，才去飞书后台验证和保存下面两个地址。

### 事件订阅 `/feishu/events`

1. 进入当前应用 -> “事件与回调”。
2. 打开“事件配置”页签。
3. 在“订阅方式”右侧点击铅笔图标。
4. 选择“将事件发送至开发者服务器”。
5. 请求地址填写：

```text
https://headhunt.zenithy.art/feishu/events
```

6. 点击“验证”或“保存”。飞书会发送 challenge；后端通过后页面才会显示成功。
7. 在“已添加事件”里添加：

```text
接收消息 v2.0 / im.message.receive_v1
```

### 卡片回调 `/feishu/card-actions`

1. 进入“回调配置”页签。
2. 请求地址填写：

```text
https://headhunt.zenithy.art/feishu/card-actions
```

3. 添加回调：

```text
卡片回传交互 / card.action.trigger
```

只配置新版 `card.action.trigger`。不要配置旧版 `card.action.trigger_v1`，也不要把卡片回调配置到 `/feishu/events`。

### 加密策略

切到“加密策略”页签，复制并记录：

- `Verification Token`
- `Encrypt Key`

这两个值必须填入服务器 `.env`：

```env
FEISHU_VERIFICATION_TOKEN=<从飞书后台复制>
FEISHU_ENCRYPT_KEY=<从飞书后台复制>
```

它们不是 `bash scripts/lietou-oneclick.sh` 自动生成的 `<auto-generated>`。如果你在飞书后台重新生成或修改，需要同步改服务器 `.env`，然后重启 API：

```bash
cd /opt/apps/headhunt-agent
nano .env
docker compose up -d --build
```

## 获取 `.env` 里的飞书字段

第一阶段必须填写这些字段：

| `.env` 字段 | 从哪里获取 | 识别方式 |
| --- | --- | --- |
| `FEISHU_APP_ID` | 凭证与基础信息 -> App ID | 通常以 `cli_` 开头 |
| `FEISHU_APP_SECRET` | 凭证与基础信息 -> App Secret | 只保存到服务器 `.env` |
| `FEISHU_VERIFICATION_TOKEN` | 事件与回调 -> 加密策略 | 与后台完全一致 |
| `FEISHU_ENCRYPT_KEY` | 事件与回调 -> 加密策略 | 不是 `MODEL_SECRET_ENCRYPTION_KEY` |
| `FEISHU_DEFAULT_CHAT_ID` | 机器人所在测试群 `chat_id` | 通常以 `oc_` 开头，不是群名称 |

第二阶段 Bitable 同步再填写：

| `.env` 字段 | 从哪里获取 |
| --- | --- |
| `FEISHU_BITABLE_APP_TOKEN` | 多维表格 URL 里 `/base/` 后面的 token |
| `FEISHU_BITABLE_REQUISITION_TABLE_ID` | 岗位/需求表的 `table=` 参数 |
| `FEISHU_BITABLE_CANDIDATE_TABLE_ID` | 候选人表的 `table=` 参数 |
| `FEISHU_BITABLE_TALENT_MAP_TABLE_ID` | 人才地图表的 `table=` 参数 |
| `FEISHU_BITABLE_REPORT_TABLE_ID` | 报告/复盘表的 `table=` 参数 |

### 获取 `FEISHU_DEFAULT_CHAT_ID`

先确认：

- 机器人已经加入目标测试群。
- 权限里已开通 `im:chat:readonly`。
- 权限已经发布到企业。
- `.env` 已填好 `FEISHU_APP_ID` 和 `FEISHU_APP_SECRET`。

然后在服务器执行：

```bash
cd /opt/apps/headhunt-agent
set -a
. ./.env
set +a

TENANT_ACCESS_TOKEN="$(
  curl -sS -X POST "https://open.feishu.cn/open-apis/auth/v3/tenant_access_token/internal" \
    -H "Content-Type: application/json" \
    -d "{\"app_id\":\"$FEISHU_APP_ID\",\"app_secret\":\"$FEISHU_APP_SECRET\"}" \
  | python3 -c "import json,sys; d=json.load(sys.stdin); assert d.get('code') == 0, d; print(d['tenant_access_token'])"
)"

curl -sS "https://open.feishu.cn/open-apis/im/v1/chats?page_size=50" \
  -H "Authorization: Bearer $TENANT_ACCESS_TOKEN" \
  | python3 -m json.tool
```

输出里找到目标群的 `name`，复制同一项里的 `chat_id`：

```env
FEISHU_DEFAULT_CHAT_ID=oc_xxxxxxxxx
```

如果输出为空或报 `access denied`，优先检查机器人是否在群里、`im:chat:readonly` 是否开通、应用版本是否已发布。

### 获取 Bitable app token 和 table ID

只有要启用 Bitable 同步时才做这一步。

打开目标多维表格，浏览器地址通常类似：

```text
https://example.feishu.cn/base/bascnxxxxxxxx?table=tblxxxxxxxx&view=vewxxxxxxxx
```

对应关系：

- `/base/` 后面的 `bascnxxxxxxxx` 是 `FEISHU_BITABLE_APP_TOKEN`。
- `table=` 后面的 `tblxxxxxxxx` 是当前表的 `table_id`。
- `view=` 后面的 `vewxxxxxxxx` 是视图 ID，不要填进 `.env`。

## 配置 `.env`

先生成 `.env` 和本地运行密钥：

```bash
cd /opt/apps/headhunt-agent
bash scripts/lietou-oneclick.sh
```

第一阶段至少填写：

```env
DOMAIN=headhunt.zenithy.art

FEISHU_BASE_URL=https://open.feishu.cn
FEISHU_APP_ID=cli_xxxxxxxxxxxxxxxx
FEISHU_APP_SECRET=<只填服务器 .env>
FEISHU_VERIFICATION_TOKEN=<从飞书加密策略复制>
FEISHU_ENCRYPT_KEY=<从飞书加密策略复制>
FEISHU_DEFAULT_CHAT_ID=oc_xxxxxxxxx
```

`INTERNAL_ADMIN_API_KEY`、`POSTGRES_PASSWORD`、`MODEL_SECRET_ENCRYPTION_KEY` 由脚本生成。不要保留字面量 `<auto-generated>`。

第二阶段启用 Bitable 再补：

```env
FEISHU_BITABLE_APP_TOKEN=bascnxxxxxxxx
FEISHU_BITABLE_REQUISITION_TABLE_ID=tblxxxxxxxx
FEISHU_BITABLE_CANDIDATE_TABLE_ID=tblxxxxxxxx
FEISHU_BITABLE_TALENT_MAP_TABLE_ID=tblxxxxxxxx
FEISHU_BITABLE_REPORT_TABLE_ID=tblxxxxxxxx
```

启动前执行：

```bash
python3 -m app.runtime.local_doctor --strict
```

第一阶段飞书字段填完后，即使 Bitable 仍为空，这个检查也可以通过；输出里可能会有 `Optional Bitable business sync is not ready`，表示审批后的多维表格同步还不能用。

## 启动和联调顺序

```bash
cd /opt/apps/headhunt-agent
docker compose up -d --build
```

先检查公网入口：

```bash
curl -f https://headhunt.zenithy.art/health
curl -f -H "X-Internal-Admin-Key: <INTERNAL_ADMIN_API_KEY>" https://headhunt.zenithy.art/ready
```

再检查飞书 challenge JSON：

```bash
set -a
. ./.env
set +a

curl -i -sS -X POST "https://headhunt.zenithy.art/feishu/events" \
  -H "Content-Type: application/json" \
  -d "{\"challenge\":\"local-check\",\"token\":\"$FEISHU_VERIFICATION_TOKEN\",\"type\":\"url_verification\"}"
```

预期返回：

```json
{"challenge":"local-check"}
```

只有这个通过后，才去飞书后台保存事件订阅 URL。

业务联调顺序：

1. 保存 `/feishu/events`，确认 challenge 通过。
2. 保存 `/feishu/card-actions`，确认卡片回调配置保存。
3. 在测试群 @机器人发送一个简单任务。
4. 检查 `feishu_event_logs` 和 `feishu_outbox(kind=card_send)`。
5. 点击任务确认卡/审批卡。
6. 第二阶段再验证 Bitable 写入和 `feishu_bitable_write_chunks`。
