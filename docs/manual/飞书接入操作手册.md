# 飞书接入操作手册

本文是第一版主入口手册。目标是把飞书机器人、事件订阅、交互卡片、War Room 群消息和 Bitable 同步接到 `/Users/w/Documents/lietou` 后端。真实飞书后台、真实 Bitable 权限、真实卡片发送和 graph 端到端联调未完成前，必须写“未验证”。

## 你需要准备

- 一个飞书开放平台应用。
- 一个测试群，机器人已被邀请进群。
- 一个可公网访问的 HTTPS 域名。
- 服务器已能执行 `docker compose up -d --build`。
- OpenAI / DeepSeek API Key 由用户后续通过飞书模型配置卡录入，不写入共享 `.env`。

## 创建飞书应用

1. 打开 [飞书开放平台](https://open.feishu.cn/)。
2. 创建企业自建应用。
3. 记录 `App ID`、`App Secret`。
4. 在“事件订阅”里记录或配置 `Verification Token` 和 `Encrypt Key`。
5. 开启机器人能力，并把机器人加入测试群。

## 配置回调地址

服务器正式地址假设为：

```text
https://<DOMAIN>
```

在飞书后台配置：

```text
事件订阅 URL: https://<DOMAIN>/feishu/events
卡片回调 URL: https://<DOMAIN>/feishu/card-actions
```

要求：

- `/feishu/events` 必须能处理 URL verification / challenge。
- `/feishu/events` 业务事件只做验签、幂等落库、outbox 和 ACK。
- `/feishu/card-actions` 只做验签、卡片动作解析、HumanApproval 或 double check 入库、outbox 和 toast。
- 不在回调请求里同步跑完整 graph、LLM、Bitable 写入或外部触达。

## 配置权限

第一版建议申请：

```text
机器人收发消息
读取群信息
发送和更新交互卡片
读取用户基础身份标识
Bitable app/table record 读写
```

权限以飞书后台实际名称为准。没有真实租户权限前，文档和测试结果只能标“未验证”。

## 配置 Bitable

1. 创建一个多维表格。
2. 为岗位、候选人、任务结果、审批结果建立展示表。
3. 记录 `app_token` 和 `table_id`。
4. 给应用授予对应 Bitable 权限。
5. 后端 `.env` 填入：

```env
FEISHU_BITABLE_APP_TOKEN=
FEISHU_BITABLE_TABLE_ID=
```

Bitable 只做业务展示和同步表。主数据、审批、outbox、AgentRuns、Artifacts、Memory 仍在 PostgreSQL。

## 配置 `.env`

复制 `.env.example` 为 `.env`，至少填写：

```env
DOMAIN=your-domain.com
INTERNAL_ADMIN_API_KEY=change-this-admin-key
MODEL_SECRET_ENCRYPTION_KEY=change-this-model-secret

POSTGRES_DB=lietou
POSTGRES_USER=lietou
POSTGRES_PASSWORD=change-this-password

FEISHU_BASE_URL=https://open.feishu.cn
FEISHU_APP_ID=
FEISHU_APP_SECRET=
FEISHU_VERIFICATION_TOKEN=
FEISHU_ENCRYPT_KEY=
FEISHU_EVENT_CALLBACK_PATH=/feishu/events
FEISHU_CARD_ACTION_CALLBACK_PATH=/feishu/card-actions
FEISHU_DEFAULT_CHAT_ID=
FEISHU_BITABLE_APP_TOKEN=
FEISHU_BITABLE_TABLE_ID=
```

不要把用户的 OpenAI / DeepSeek Key 写进 `.env` 作为共享业务模型。第一版目标是用户通过飞书卡片 BYOK。

## 启动服务

在项目根目录执行：

```bash
docker compose up -d --build
docker compose ps
curl https://<DOMAIN>/health
curl -H "X-Internal-Admin-Key: $INTERNAL_ADMIN_API_KEY" https://<DOMAIN>/ready
```

本机没有 Docker 时，不能写成已通过；只记录“未验证”并在服务器执行。

## 联调顺序

1. 先保存 `/feishu/events`，确认 challenge 通过。
2. 再保存 `/feishu/card-actions`，确认签名校验通过。
3. 在测试群 @机器人发送一个简单任务。
4. 后端应 ACK 并入 `feishu_event_logs + feishu_outbox`。
5. 系统发送任务确认卡。
6. 用户 approve/edit/reject。
7. approve 后进入 graph；高风险动作继续通过飞书确认卡走 HumanApproval。
8. 人工确认后才同步 Bitable。

## 模型配置

第一版目标是通过飞书模型配置卡：

```text
provider: openai / deepseek
usage: chat / embedding
model_name
display_name
api_key
base_url 可选
```

API Key 由 `MODEL_SECRET_ENCRYPTION_KEY` 加密保存，不在飞书卡片、日志、AgentRuns 或查询接口展示。

## 未验证边界

- 真实飞书后台保存回调未验证。
- 真实飞书卡片发送/更新未验证。
- 真实 Bitable 写入未验证。
- 真实 OpenAI/DeepSeek 调用未验证。
- 真实 graph 端到端联调未验证。
