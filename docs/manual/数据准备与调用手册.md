# 数据准备与调用手册

## 状态

当前已有 Phase 2 工程骨架、Pydantic schema 和 PostgreSQL 表模型。本手册定义
第一版系统需要的数据格式和调用方式；导入类 API、真实 Discord War Room 调用和 examples
目录仍待后续阶段实现后用 OpenAPI 继续校准。

## 最小可用数据

第一版跑通至少需要：

- 1 条岗位需求。
- 3-10 条候选人摘要。
- 3-5 条历史案例或用户修正，作为长期记忆审批样例。
- 1 个 Discord Server 和允许使用的 channel。
- 1 个 PostgreSQL + pgvector 数据库。
- 能标识数据范围的 scope 字段：`tenant_id`、`guild_id`、`user_id`、`project_id`、`requisition_id`、`candidate_id`。
- 公司当前事实的 `SearchGateway source_refs`，例如官网、新闻、融资公告、招聘页或可信检索结果。

## 结构化字段来源规范

`TaskIntakeParser` 输出的关键字段必须有来源和置信度。人工准备数据时也应尽量提供：

```json
{
  "field_name": "current_company",
  "value": "某 AI 公司",
  "field_source": {
    "kind": "resume_artifact",
    "ref": "artifact://resume-source/xxx"
  },
  "confidence": 0.86
}
```

规则：

- 有来源、有置信度的字段才能进入 `facts`。
- 无来源推断只能进入 `assumptions`，不能写进岗位事实、候选人事实或公司当前事实。
- 用户在 Discord double check 点 Approve 后，`CanonicalTaskBrief.version` 冻结；后续修改必须通过 Edit 生成新版本并再次确认。
- 下游 Agent 只能读取冻结版 `CanonicalTaskBrief`、ArtifactRef、MemoryRef、SOPRef 和 SourceRef。

## 历史案例 / 长期记忆输入

长期记忆不能直接 active，需要审批。

```json
{
  "scope": "CaseMemory",
  "summary": "AI 平台负责人岗位中，模型平台和工程平台职责混合时，应先拆清平台边界。",
  "content_ref": "artifact://case-memory-source/v1",
  "owner_agent": "StrategyDraftAgent",
  "confidence": 0.8,
  "metadata": {
    "tenant_id": "tenant-default",
    "guild_id": "discord-guild-123",
    "user_id": "discord-user-456",
    "project_id": "project-ai-platform",
    "requisition_id": "REQ-001",
    "candidate_id": null,
    "domain": "AI平台",
    "role_family": "后端负责人"
  }
}
```

如果用户上传原文，接口必须先把原文写入 ArtifactStore 或 Memory blob store，返回 `content_ref`；ContextPack 只注入 `summary` 和 ref，不注入全文。

MemoryGateway 检索时必须先按 `thread_id / tenant_id / guild_id / user_id / project_id / requisition_id / candidate_id` 做 scope filter，再做向量召回、rerank、MMR 去重和 token budget 裁剪。长期记忆不得替代实时搜索。

流程：
1. `MemoryGateway.propose_update` 生成 `draft` 或 `pending_review` memory。
2. 人工在 War Room 确认。
3. 审批通过后写入 `status=active`，并设置 `retention_policy`，例如 `90d`。
4. EmbeddingGateway 向量化。
5. 后续任务按权限和相关性命中为 `MemoryRef`。

Retention 建议：
- RunMemory 默认 30 天。
- ProjectMemory / AgentMemory / CaseMemory / UserCorrectionMemory 默认 90 天。
- 重要记忆可人工续期或升级为 permanent。

当前事实边界：

- 公司背调、当前在职状态、融资、新闻、裁员和组织变化必须来自 SearchGateway 的 `source_refs`。
- 长期记忆只提供 taxonomy、项目规则、用户修正、历史案例和 SOP。
- 简历关键词抽取以当前简历 artifact 和当前 JD artifact 为主；长期记忆只能辅助行业词表、项目偏好和用户修正规则。

## API 调用示例

### 导入岗位

```bash
curl -X POST http://127.0.0.1:8000/imports/requisitions \
  -H "Content-Type: application/json" \
  -d @examples/requisition.json
```

### 导入候选人

```bash
curl -X POST http://127.0.0.1:8000/imports/candidates \
  -H "Content-Type: application/json" \
  -d @examples/candidates.json
```

### 生成人才地图

```bash
curl -X POST http://127.0.0.1:8000/talent-map/run \
  -H "Content-Type: application/json" \
  -d '{
    "requisition_id": "uuid",
    "request_text": "请生成第一版人才地图，优先华东 AI 平台公司",
    "require_human_approval": true
  }'
```

### 候选人筛选

```bash
curl -X POST http://127.0.0.1:8000/candidates/uuid/screen \
  -H "Content-Type: application/json" \
  -d '{
    "requisition_id": "uuid",
    "request_text": "判断这个候选人是否适合进入推荐名单"
  }'
```

## Discord 调用方式

在允许的 Discord channel 里使用 slash command：

```text
/headhunt new role_title:<岗位名称> jd_text:<岗位描述> output_goal:<希望输出>
```

如果你要强制完整会审：

```text
/headhunt new role_title:<岗位名称> jd_text:<岗位描述> council_mode_hint:三省六部
```

系统必须在卡片里显示：

```text
会审模式: full_council
原因: 用户明确要求“三省六部”
```

任务确认卡必须展示关键字段的 `field_source`、`confidence` 和 `assumptions`。确认卡 Approve 后冻结 `CanonicalTaskBrief.version`；Edit 会生成新版本并要求再次确认；未 Approve 不得启动正式 graph。

## 输出你会看到什么

- 任务确认卡 double check。
- 进度卡。
- 追问卡。
- 确认卡。
- 结果卡。
- 每张卡展示 `council_mode`、`mode_reason`、artifact refs、memory refs、sop refs、ReviewGate 结果和 token 估算。
- ReviewGate 结果只针对当前 artifact：`pass` 进入下一步，`needs_fix` 只回对应 repair_node 一次，第二次仍失败必须进入人工确认。

## 人工确认场景

你需要在 Discord button / modal 里确认：

- 是否写入岗位校准结果。
- 是否写入人才地图。
- 是否发布候选人报告。
- 是否创建外部任务。
- 是否发送外部触达。
- 是否提交推荐结论。
- 是否把复盘写成长期记忆。
