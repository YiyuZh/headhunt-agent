# 接口总览

## 状态

本文是接口施工文档。当前代码已实现 `GET /health`、受保护的 `GET /ready`、`GET /version`、`POST /council/deliberate`、`POST /tasks/authorize`、Feishu 主路径 `POST /feishu/events` 和 `POST /feishu/card-actions`、`GET /threads/{thread_id}`、`GET /runs/{run_id}`、`GET /approvals/{interrupt_id}`、`POST /approvals/{interrupt_id}/decision`。

本轮 PRD 已切换为 Feishu First。第一版普通用户入口是 `/feishu/events` 和 `/feishu/card-actions`；飞书卡片承担任务 double check、approve/edit/reject、模型 BYOK 配置和 graph resume。当前 Feishu verifier、FeishuGateway、BitableGateway、feishu_outbox 和审批 resume 代码已有基础；真实飞书后台、Bitable 权限、真实卡片发送和 graph 端到端联调仍未验证。Discord `/discord/interactions`、`/model` 和 command register 仅作为 optional adapter / 历史实现保留。

## API 分组

### 系统健康

| 方法 | 路径 | 用途 | 认证 | 副作用 | 状态 |
| --- | --- | --- | --- | --- | --- |
| GET | `/health` | 进程健康检查 | 无 | 无 | 已实现 |
| GET | `/ready` | 数据库、checkpointer、outbox、Feishu/Discord 配置就绪检查 | 内部 admin key | 无 | 已实现；飞书真实联调未完成时显示 warning |
| GET | `/version` | 返回版本、git sha、配置摘要 | 内部 admin key | 无 | 已实现 |

除 `/health` 和外部平台回调外，当前内部控制面 API 都必须带 `X-Internal-Admin-Key` 或 `Authorization: Bearer <INTERNAL_ADMIN_API_KEY>`。`/ready` 只证明配置项是否齐全，不证明真实飞书权限、用户 BYOK 模型 profile、真实 OpenAI/DeepSeek Key、真实 Bitable 权限或真实 PostgreSQL 网络连通已经通过。

### 飞书入口

| 方法 | 路径 | 用途 | 响应策略 | 状态 |
| --- | --- | --- | --- | --- |
| POST | `/feishu/events` | 飞书事件订阅、URL verification、机器人消息入口 | 验签、challenge、幂等落库、feishu_outbox、3 秒 ACK | 已实现基础；真实飞书后台未验证 |
| POST | `/feishu/card-actions` | 飞书卡片 approve/edit/reject、模型配置、HumanApproval resume | 验签、幂等落库、toast、feishu_outbox | 已实现基础；真实卡片未验证 |
| POST | `/internal/feishu/outbox/dispatch` | 内部调度 feishu_outbox | worker 使用，不暴露公网 | 已实现基础 |

`/feishu/events` 与 `/feishu/card-actions` 路由规则：

```text
URL verification -> 立即返回 challenge。
用户消息 / @机器人 -> ACK -> TaskIntakeParser -> 飞书任务确认卡。
模型配置卡 -> add/list/use/test/revoke -> 用户 BYOK profile 加密保存/选择。
task double check 卡片 -> approve/edit/reject。
approval 卡片 -> HumanApproval -> Command(resume=...)。
status 卡片或内部查询 -> 查询 thread/run 摘要并返回飞书卡片。
```

### Task Double Check

任务启动必须先经过结构化确认。

| 入口 | 用途 | 状态变化 |
| --- | --- | --- |
| 飞书消息 / 入口卡 | 新岗位 / 新任务 | `pending_user_check` |
| 飞书候选人卡 | 候选人筛选 | `pending_user_check` |
| 飞书 Mapping 卡 | 人才地图 | `pending_user_check` |
| `task_check:approve:{task_id}` | 用户确认结构化任务无误 | `approved`，启动 graph |
| `task_check:edit:{task_id}` | 打开飞书卡片表单修改字段 | `edited`，重新解析并再次确认 |
| `task_check:reject:{task_id}` | 拒绝任务 | `rejected`，停止任务 |

任务确认卡必须展示：

```text
任务类型
用户目标
岗位摘要
候选人摘要
输出目标
缺失字段
风险等级
推荐 council_mode 和 mode_reason
系统假设
关键字段的 field_source 和 confidence
```

任务确认规则：
- 无 `field_source` 的字段只能展示在 assumptions，不得进入事实字段。
- approve 后冻结 `CanonicalTaskBrief.version`，后续 graph 只读取冻结版。
- edit 必须生成新 version 并重新发送确认卡。
- reject / expired 不得启动正式 graph。

### 会审与任务

| 方法 | 路径 | 用途 | 关键响应字段 | 状态 |
| --- | --- | --- | --- | --- |
| POST | `/council/deliberate` | 内部/调试运行需求会审 | `council_mode`, `mode_reason`, `decision`, `required_agents`, `optional_agents` | 已实现 |
| POST | `/tasks/authorize` | 内部调试授权一个 War Room 任务，并在 `approved=true` 时排队 `graph_dispatch` | `thread_id`, `task_id`, `source_ref`, `council_mode`, `idempotency_key`, `outbox_payload_ref`, `next_actions` | 已实现，需迁移到 Feishu outbox 主链路 |
| GET | `/threads/{thread_id}` | 查看线程状态摘要 | `state_summary`, `artifact_refs`, `memory_refs`, `pending_interrupts` | 已实现 |
| GET | `/runs/{run_id}` | 查看 AgentRun 审计摘要 | `agent_name`, `context_pack_ref`, `token_estimate`, `status` | 已实现 |

ReviewResult 路由：

## ACK / Outbox 规则

- ACK 前只做签名、allowlist、幂等、payload 持久化和 outbox 入队。
- ACK/toast 成功不代表 graph 完成，只代表后台任务已排队。
- 飞书长期消息和卡片更新统一通过 FeishuGateway + durable outbox。
- 所有 outbox worker 必须原子 claim，重复 idempotency_key 不能重复执行 graph 或 resume。
- graph、LLM、搜索、数据库写入、Bitable 写入、外部触达都不能发生在飞书回调 HTTP 请求内。
