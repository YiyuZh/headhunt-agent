# AI 猎头作战系统实施总路线图

## 目标

本路线图把现有 PRD 拆成可连续执行的工程阶段。Codex 后续执行时必须按阶段推进：先生成设计与接口文档，再生成工程骨架，再逐步实现数据库、Gateway、Discord War Room、LangGraph、AgentHarness、ReviewGate、SOPRegistry、长期记忆、测试和操作手册。每一阶段完成后必须调用相应审查 Agent 或技能复核，发现问题先修复，再进入下一阶段。

本文件不替代 PRD；PRD 仍是产品口径来源。本文件负责把 PRD 变成施工顺序、质量门和人工接入步骤。

## 固定约束

- 工作目录固定为 `/Users/w/Documents/lietou`。
- 第一版主链路使用 FastAPI、LangGraph、PostgreSQL、pgvector、真实 Discord interactions、Discord buttons、select menus、modals 和 War Room threads/messages。
- 飞书/Bitable 是 deferred adapter，不进入第一版主验收路径。
- mock / fake 只用于单测、CI 和失败隔离，不作为第一版主运行路径。
- Discord `/discord/interactions` 只做 signature 校验、幂等、落库、排队和 3 秒内 ACK/defer；完整 graph 由异步 worker 执行。
- 用户必须 double check `CanonicalTaskBrief` 等结构化任务后，正式 `headhunter_war_room_graph` 才能启动；approve 后冻结 version。
- `TaskIntakeParser` 每个关键字段必须有 `field_source` 和 `confidence`，无 source 推断只能进入 assumptions。
- 运行图统一为 `headhunter_war_room_graph -> council_deliberation_graph -> business subgraph`。
- 三省六部是同一 Canonical Context 下的角色审查矩阵，不是自由 agent-to-agent 聊天。
- Agent 不直接互相调用，不直接访问公网、数据库、Discord、飞书或模型 Key；所有外部能力必须通过 Gateway、Registry 和 PolicyEngine。
- AgentHarness 只给 Agent 注入 `ContextPack`，不得塞完整聊天历史、完整 `RecruitmentState`、完整 `node_history`、全量 AgentRuns、全量 artifacts 或全量长期记忆。
- 所有用户可见响应和 War Room 卡片必须展示本次 `council_mode` 和 `mode_reason`。
- 用户输入“三省六部”时强制 `full_council`；否则由 PolicyEngine / AgentHarness 自动选择 `triage | lite | standard | full_council`。
- 任务授权后的 Discord War Room 进度卡、追问卡、确认卡和结果卡可以自动发送；业务数据写入、报告发布、外部触达发送、候选人推荐结论和长期记忆 active 必须 interrupt 人工确认。
- 记忆第一版使用 PostgreSQL + pgvector；长期记忆按 `30d / 90d / permanent` retention policy 管理。
- MemoryGateway 必须按 tenant/guild/user/project/requisition/candidate scope filter 检索；公司当前事实必须由 SearchGateway source_refs 支撑。

## 阶段计划

### Phase 0: 文档施工图

目标：生成工程实施必须先有的接口、数据库、Discord、Agent、测试和操作文档。

输出：
- `docs/engineering/01_接口总览.md`
- `docs/engineering/02_数据库设计.md`
- `docs/engineering/03_Discord接入设计.md`
- `docs/engineering/03_飞书接入设计.md`，标记 deferred adapter
- `docs/engineering/04_Agent与ContextPack设计.md`
- `docs/engineering/05_测试验收计划.md`
- `docs/manual/本地部署与使用手册.md`
- `docs/manual/Discord接入操作手册.md`
- `docs/manual/飞书接入操作手册.md`，标记 deferred adapter
- `docs/manual/数据准备与调用手册.md`
- `docs/manual/猎头日常使用手册.md`
- `docs/manual/故障排查手册.md`

质量门：
- Markdown 文件覆盖接口、表、Discord 接入、Agent 上下文、ReviewGate、SOPRegistry、测试、部署、人工操作、数据输入和日常使用。
- 文档明确“已实现 / 未实现 / 未验证”状态。
- 索引文件和任务记忆同步更新。
- 审查 Agent 返回的问题已处理，或明确记录为下一阶段事项。

### Phase 1: 工程骨架

目标：创建可启动的 Python 工程骨架，但只实现健康检查、配置加载和依赖方向。

输出：
- `pyproject.toml`
- `.env.example`
- `app/main.py`
- `app/api/*`
- `app/core/config.py`
- `app/schemas/*`
- `tests/*`

质量门：
- `ruff` 和基础 `pytest` 可运行。
- `.env.example` 默认使用 PostgreSQL、pgvector、Discord 配置占位和 deferred Feishu 配置占位。
- 不出现 SQLite 作为主路径。

### Phase 2: 数据库与迁移

目标：实现 PostgreSQL、pgvector、SQLAlchemy models、Alembic migrations、repositories 和 checkpointer 连接。

输出：
- `app/storage/models/*`
- `app/storage/repositories/*`
- `migrations/*`
- `docker-compose.yml`
- PostgreSQL + pgvector 初始化脚本。

质量门：
- 迁移可在本地 PostgreSQL 跑通。
- 所有外键、幂等键、常用查询字段、vector 检索字段都有索引或约束。
- `discord_event_logs`、`discord_outbox`、`discord_interactions`、`discord_message_map`、`discord_installations`、AgentRuns、Artifacts、MemoryItems、MemoryEmbeddings 表进入主链路。

### Phase 3: 核心接口与 Policy

目标：实现 Pydantic schemas、PolicyEngine、ActionProposal、HumanApproval、ArtifactStore、ContextPack、MemoryGateway、AgentSOPRegistry 和 ReviewGate 接口。

质量门：
- Agent 无法绕过 Gateway 访问外部资源。
- `AgentArtifact` 只存摘要和 `content_ref`。
- `MemoryGateway.retrieve` 默认只返回 `MemoryRef`，不返回全文。
- `MemoryGateway.retrieve` 必须做 tenant/guild/user/project/requisition/candidate scope filter。
- `ContextPack` 包含 `sop_refs`，但不注入全部 SOP。
- ReviewGate 是 artifact-level quality gate，具备 `pass | needs_fix | needs_human` 三态并通过 conditional edge 路由。

### Phase 4: LangGraph 与 AgentHarness

目标：实现 double check 后的根图、council 子图、业务子图和最小上下文注入。

质量门：
- `CanonicalTaskBrief` 未经用户 double check 不得启动正式 graph；approve 后冻结 version，下游只读冻结版。
- `council_mode` 自动选择和用户强制 `full_council` 都有测试。
- 每个 Agent 的 Context allowlist 有单测。
- CouncilSynthesizer 只拿各 Agent 结论摘要，不拿原始 ContextPack。
- ReviewGate 的 `needs_fix` 只回对应 repair_node 一次，不重跑完整 graph；第二次失败必须人工确认。

### Phase 5: Discord 真实接入

目标：实现真实 Discord interaction、任务确认卡、审批按钮、modal 编辑、War Room message/thread、幂等 outbox。

质量门：
- `/discord/interactions` signature 校验通过，PING 响应正确。
- `/discord/interactions` 3 秒 ACK/defer，不同步跑 graph。
- ACK 前 PostgreSQL 事务必须提交；DB commit 失败不得 ACK 成功，应让 Discord 或人工重试。
- 重复 interaction_id / idempotency_key 不重复创建 graph run 或 resume。
- task double check approve/edit/reject 全链路可测。
- button / modal 能生成 HumanApproval 并异步 `Command(resume=...)`。
- Discord follow-up token 15 分钟限制和 bot token 长期消息更新路径写清楚。
- 真实 Discord 联调未执行前必须写“未验证”。

### Phase 6: 业务闭环

目标：实现需求会审、人才地图、候选人筛选、报告草稿、复盘记忆的第一版可用链路。

质量门：
- 人工确认边界和 interrupt/resume 跑通。
- 自动 War Room 卡片展示 `council_mode`、memory_refs、sop_refs、artifact_refs、ReviewGate 结果和 token 估算。
- 业务写入、报告发布、外部触达和推荐结论不越过人工确认。

### Phase 7: 记忆与 SOP 一步到位

目标：实现 PostgreSQL + pgvector 的 RunMemory、长期记忆审批、30/90/permanent retention、SOPRegistry。

质量门：
- RunMemory 自动写入并向量化。
- 长期记忆未经审批不可 active。
- 过期、撤销、pending_review 记忆不可检索。
- retention job 能处理 30d / 90d / permanent。
- 公司当前事实不得由长期记忆替代实时搜索；SearchGateway source_refs 是当前事实依据。
- 简历/JD 关键词抽取以当前 artifact 为主，长期记忆只提供 taxonomy、项目规则、用户修正和 SOP。
- SOPRegistry 支持 Always / Auto Attached / Agent Requested / Manual 触发模型。

### Phase 8: 测试、审查和调优

目标：补齐单测、集成测试、Discord 契约测试、ContextPack 预算测试、安全测试和审查 Agent 复核。

质量门：
- 测试覆盖核心路径。
- 审查 Agent 返回“无阻断问题”或所有阻断问题已修复。
- 未联调真实外部服务的事项明确写“未验证”。

### Phase 9: 使用手册和交付

目标：更新操作手册，让用户知道怎么配置 Discord、给什么数据、怎么调用、能做什么、哪里需要人工确认。

质量门：
- 手册可支持从零启动。
- Discord Developer Portal 人工步骤清晰。
- API 调用示例和数据模板可复制执行。
- 飞书手册明确 deferred adapter，不误导为第一版主入口。

## 每阶段执行循环

1. 读取 `任务记忆文档.md`、`项目文件索引.md`、`项目文件索引.json` 和当前阶段相关 PRD。
2. 必要时上网查看官方文档，优先官方来源。
3. 生成或修改本阶段文件。
4. 运行格式、JSON、测试或可行的静态检查。
5. 调用相应审查 Agent 复核架构、Discord、数据库、测试或 token 消耗问题。
6. 按审查问题修复。
7. 更新任务记忆和文件索引。
8. 进入下一阶段。
