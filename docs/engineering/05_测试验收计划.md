# 测试验收计划

## 状态

当前项目已有 Phase 14 内部人工审批 API、Phase 13 只读审计查询 API、Phase 12 任务授权 API、Phase 11 外部服务
smoke 检查入口和测试集；当前飞书 `/feishu/events`、`/feishu/card-actions`、FeishuGateway、
FeishuBitableGateway 和 outbox handler 已有基础测试；Discord `/discord/interactions` 的 PING + `/model` BYOK
作为 optional adapter 保留单元测试。最近一次验证为 `.venv/bin/python -m pytest -q`
196 passed, 2 warnings；`.venv/bin/python -m ruff check .` 通过。真实 PostgreSQL 在线迁移、
真实 PostgreSQL checkpointer、真实 OpenAI/DeepSeek 调用、真实飞书后台、真实 Bitable 权限、
War Room 卡片和 graph 端到端联调仍未验证。

命令：

```bash
python3 -m pytest

```

- FastAPI `/health` 可启动返回成功。
- `.env.example` 默认 PostgreSQL、pgvector、Feishu、Embedding、LangGraph checkpoint 配置，Discord 只作为 optional adapter 配置。
- 不把 SQLite 写入主路径配置。
- fake/mock 目录只用于 tests 或 gateway fakes。

## Phase 3 验收

要求：
- `AgentArtifact` 不再有旧式全文 dict 字段进 state 的实现。
- `MemoryGateway.retrieve` 返回 `MemoryRef`，默认不返回全文。
- `ActionProposal` 和 `HumanApproval` 有 `interrupt_id`、`action_id`、`idempotency_key`、`approver`、`decision`、`edited_payload`。
- PolicyEngine 能阻止越权 Gateway、越权 memory scope 和越权副作用。
- AgentSOPRegistry 只能返回 policy 允许的 SOPRef。
- ReviewGate 能输出 `pass | needs_fix | needs_human`。
- ReviewGate 是 artifact-level gate；测试中不得读取完整聊天历史、完整 state、其他 Agent 原始 ContextPack 或原始 prompt。
- ReviewGate conditional edge 能按 `pass / needs_fix / needs_human` 路由。
- `needs_fix` 只回对应 repair_node 一次，不重跑完整 graph；第二次仍 `needs_fix` 必须进入 `interrupt_human_approval`。

## Phase 4 验收

要求：
- `council_mode` 自动选择测试：
  - 简单任务 -> `lite`
  - 缺信息 -> `triage`
  - 常规猎头任务 -> `standard`
  - 用户要求“三省六部” -> `full_council`
- 所有用户响应和 War Room 卡片包含 `council_mode` 和 `mode_reason`。
- `full_council` 必须运行六部并行 + `ChallengeReviewAgent + CouncilSynthesizerAgent`，六部意见全部进入 `department_opinions` 并写入 ArtifactStore。
- 每个 Agent 的 ContextPack 不包含完整 state/history/node_history/AgentRuns/all artifacts/all memories。
- `ContextPackBuilder` 不接收 raw `RecruitmentState`、raw messages、raw `node_history` 或完整 AgentRuns，只接收 typed selector / allowed snapshot。
- LLMGateway 测试断言只收到 `ContextPack` 和 schema，不能收到 raw state、messages、node_history 或其他 Agent 原始 ContextPack。
- CouncilSynthesizer 不读取原始 ContextPack。
- 正式 graph 只能从 double checked `CanonicalTaskBrief` 启动。
- `CanonicalTaskBrief` approve 后冻结 version，下游 Agent 只能读取冻结版。
- TaskIntakeParser 每个关键字段必须有 `field_source` 和 `confidence`；无 source 推断只能进入 `assumptions`，不能进入 facts。
- 下游 Agent 不得默认重新解析原始任务或原始简历。

## Phase 5 验收：Feishu First

要求：
- `/feishu/events` URL verification 返回 challenge。
- `/feishu/events` 与 `/feishu/card-actions` 校验飞书签名、timestamp、nonce、verification token/encrypt key。
- 飞书模型配置卡覆盖 add/list/use/test/revoke 目标流程，响应不得返回明文或密文 API Key。
- 飞书回调在 3 秒内 ACK/toast，只做验签、allowlist、轻量卡片处理或 outbox，不同步跑 graph。
- ACK 前 PostgreSQL 事务必须提交；DB commit 失败不得 ACK 成功。
- 模拟 slow worker / slow LLM / slow graph 时，ACK 仍不被阻塞。
- 重复 `event_id` 或 `action_id` 不重复创建 graph run。
- 重复 `idempotency_key` 不重复 resume。
- `/headhunt new` / `/headhunt candidate` / `/headhunt map` 能生成 `CanonicalTaskBrief`。
- 任务确认卡 approve/edit/reject 都有测试。
- edit 卡片表单修改字段后必须重新解析并再次发送确认卡。
- edit 后必须生成新的 `CanonicalTaskBrief.version`；旧冻结版不得原地修改。
- double check 未 approve 不得启动正式 graph。
- approval 卡片能生成 `HumanApproval` 并排队 `resume`。
- `edit` 审批必须带 `edited_payload`。
- 飞书消息和卡片更新统一走 FeishuGateway + outbox。
- 真实飞书联调未执行前必须写“未验证”。

## Phase 5b 验收：Discord Optional Adapter

要求：
- `/feishu/events` URL verification 目标 1 秒内原样返回 challenge；加密场景先验签/解密再返回。
- `/feishu/events` 对业务事件在 3 秒内 ACK，只做验签、幂等落库和 outbox，不同步跑 graph。
- ACK 前 PostgreSQL 事务必须提交；DB commit 失败不得 ACK 成功，应让飞书重试。
- 模拟 slow worker / slow LLM / slow graph 时，ACK 仍不被阻塞。
- 重复 `event_id` 不重复创建 graph run。
- 同一 `im.message.receive_v1` 的不同 `event_id` 但相同 `message_id` 不重复创建 graph run。
- `/feishu/card-actions` 在 3 秒内响应 toast，对重复 `idempotency_key` 不重复 resume。
- Bitable `batch_create` 分片必须传持久化 `client_token`，网络超时重试不得重复创建记录，client_token conflict 进入审计/人工分支。
- token 缓存刷新、401、429、Retry-After、dead letter 有测试。
- Bitable 分片写入、upsert 和 record_id 映射有测试。
- graph_dispatch / resume 在未接真实 AgentHarness / interrupt 时不得假成功，必须进入 retry/dead-letter 路径。
- 本阶段只保护历史 adapter，不作为第一版主链路验收 blocker。

## Phase 8 验收

要求：
- 覆盖预算测试：超预算先裁剪低相关 memory，再裁剪 optional source_refs，不注入全历史。
- 覆盖长期记忆审批测试：`draft/pending_review` 不可检索，`active` 可检索，`revoked/expired` 不可检索。
- 覆盖 MemoryGateway 缓存失效测试：记忆撤销、过期、审批状态变化、PII 权限降低、scope 变更或 policy version 变化后，缓存不得继续返回旧 `MemoryRef`。
- 覆盖安全测试：CandidateJudgementAgent 不得读取联系方式明文记忆；CouncilSynthesizer 不得读取原始 ContextPack。
- 覆盖 SOP 注入测试：每个节点最多 1 个主 SOP + 2 个审查 SOP，不能注入全部 SOP 库。
- 覆盖 ReviewGate 测试：schema/practicality/context budget/safety 分别可触发 `needs_fix` 或 `needs_human`。
- 覆盖 ReviewGate repair 测试：第一次 `needs_fix` 进入指定 repair_node；第二次 `needs_fix` 进入人工确认；不得触发完整 graph 重跑。
- 覆盖 retention 测试：RunMemory 30 天、长期记忆 90 天、permanent 可撤销但不过期。
- 覆盖 MemoryGateway scope filter：thread_id / tenant_id / guild_id / user_id / project_id / requisition_id / candidate_id 缺失或不匹配时不得命中越权记忆。
- 覆盖 SearchGateway 边界：公司背调当前事实必须有 SearchGateway `source_refs`；长期记忆不得替代实时搜索。
- 覆盖简历/JD 抽取边界：关键词抽取以当前简历/JD artifact 为主，长期记忆只提供 taxonomy、项目规则、用户修正和 SOP。
- Phase 7 只读审查 Agent 已返回 interrupt/resume 风险点；已据此修复 ActionGate 并发幂等、
  确认卡单 action 单 key、`interrupt_id` 必填、pending ActionProposal 校验和 Bitable
  写入审批校验。
