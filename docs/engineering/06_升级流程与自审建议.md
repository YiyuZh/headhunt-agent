# 升级流程与自审建议

## 文档目的

本文基于当前代码和文档状态，给出 Feishu First 后续升级流程、每步自审问题和验收要求。它不替代 PRD；未真实联调事项必须继续标注“未验证”。

## 当前基线

已经具备：

- FastAPI 后端工程骨架、Pydantic schema、SQLAlchemy、Alembic、PostgreSQL + pgvector 表结构。
- Docker Compose 部署底座：`api`、`worker`、`postgres(pgvector)`、`caddy`。
- 飞书基础：`/feishu/events`、`/feishu/card-actions`、Feishu verifier、challenge、FeishuEventLog、FeishuCardAction、HumanApproval、feishu_outbox、FeishuGateway、FeishuBitableGateway、Bitable chunk / record_id 映射和审批 resume 基础。
- AgentHarness、LLMGateway、EmbeddingGateway、ArtifactStore、MemoryGateway、ActionGate、interrupt/resume、本地 graph 测试路径。
- 内部控制面已受 `INTERNAL_ADMIN_API_KEY` 保护。
- Discord `/discord/interactions` PING、`/model` BYOK 和 command register 代码保留为 optional adapter。

尚未具备或未验证：

- 真实飞书后台配置、真实 Bitable 权限、真实卡片发送、真实 graph 端到端联调。
- 飞书任务入口卡、TaskIntakeParser double check、飞书 War Room 群消息和卡片更新。
- 飞书卡片版 BYOK 模型配置。
- ReviewGate 运行时、AgentSOPRegistry 运行时、memory retention job、SearchGateway 当前事实检索。
- 真实 OpenAI / DeepSeek / embedding 调用、真实 PostgreSQL checkpointer interrupt/resume、服务器 `docker compose up -d --build`。

## 升级原则

1. 先补飞书用户入口，再补 Agent 智能增强。
2. `/feishu/events` 和 `/feishu/card-actions` 只做验签、幂等、落库、outbox 和 3 秒内 ACK/toast，不同步跑 graph。
3. 用户未 double check approve 前，不得启动正式 graph。
4. 下游 Agent 只读冻结版 `CanonicalTaskBrief`、ArtifactRef、MemoryRef、SOPRef、SourceRef。
5. ReviewGate 是 artifact-level quality gate，不是全局聊天审查。
6. BYOK 模型 key 默认只属于当前飞书租户/用户，不做无授权共享。
7. Bitable 只做人工确认后的业务展示和同步表，PostgreSQL/pgvector 仍是主库和记忆底座。
8. 每阶段结束必须执行测试、审查和文档同步。

## 升级阶段

### Upgrade 0：状态冻结与迁移预检

目标：确保升级前工作区可测试，文档中的未验证边界准确。

验收：

```bash
python3 -m json.tool 项目文件索引.json
.venv/bin/python -m ruff check .
.venv/bin/python -m pytest -q
sh -n scripts/docker-entrypoint.sh
```

自审问题：

- `.env.example` 与 Docker Compose 是否仍把业务模型选择放在 `.env` 主路径？
- 文档是否仍把 Discord 写成第一版主入口？
- 飞书真实联调未做的事项是否标注“未验证”？

### Upgrade 1：飞书后台与回调配置

目标：让飞书后台能保存事件订阅和卡片回调地址。

产出：

- 飞书应用、机器人、事件订阅、卡片回调和权限清单。
- `https://<DOMAIN>/feishu/events` 和 `https://<DOMAIN>/feishu/card-actions` 配置步骤。
- tenant/chat allowlist 配置方式。

验收：

- URL verification / challenge 单测通过。
- 缺签名、签名错误、timestamp 过期请求被拒绝。
- 真实飞书后台未保存前标注“未验证”。

### Upgrade 2：飞书 event/card outbox

目标：所有飞书业务事件和卡片动作先落库，再异步处理。

产出：

- `feishu_event_logs`、`feishu_card_actions`、`feishu_outbox` 幂等路径。
- `event_id/message_id/action_id/idempotency_key` 去重策略。
- worker claim、retry、dead letter 和可读错误。

验收：

- ACK/toast 前 PostgreSQL 事务已提交。
- 重复事件、重复卡片点击、重复 resume 不重复启动 graph 或执行副作用。
- slow worker / slow LLM / slow graph 不阻塞飞书回调。

### Upgrade 3：TaskIntakeParser 与飞书 Double Check

目标：飞书输入先解析为结构化任务，并让用户确认。

产出：

- `CanonicalTaskBrief`、`RequisitionBrief`、`CandidateEvidencePack` schema。
- 每个关键字段都有 `field_source` 和 `confidence`。
- 飞书任务确认卡 approve/edit/reject。

验收：

- approve 后冻结 `CanonicalTaskBrief.version`。
- edit 生成新版本并重新发确认卡。
- reject / expired 不启动 graph。

### Upgrade 4：飞书 War Room 与审批卡

目标：把进度、追问、ReviewGate、人工确认和结果展示到飞书 War Room。

产出：

- FeishuGateway 发送/更新消息和卡片。
- War Room 群消息、进度卡、追问卡、确认卡、结果卡。
- HumanApproval 卡片 approve/edit/reject 到 `Command(resume=...)`。

验收：

- War Room 展示 `council_mode`、`mode_reason`、artifact_refs、memory_refs、sop_refs 和 ReviewGate。
- Bitable 写入、推荐结论、报告发布、外部触达必须先 HumanApproval。
- 卡片限频、权限不足、消息更新失败进入 retry/dead letter，不让 Agent 绕过 gateway。

### Upgrade 5：Bitable 同步

目标：人工确认后把业务展示数据同步到 Bitable。

产出：

- Bitable table registry、chunk 写入、client_token、record_id 映射。
- `ActionProposal -> HumanApproval -> feishu_outbox -> FeishuBitableGateway` 主链路。

验收：

- 未审批不得写 Bitable。
- 网络重试不重复创建记录。
- client_token payload conflict 进入审计/人工分支。

### Upgrade 6：ReviewGate、SOPRegistry 与记忆治理

目标：补齐 AI 工作流质量门和长期记忆治理。

验收：

- ReviewGate 按 `pass / needs_fix / needs_human` conditional edge 路由。
- `needs_fix` 只回对应 repair_node 一次，第二次失败进入人工确认。
- SOP 只以 `sop_refs` 注入，每节点最多 1 个主 SOP + 2 个审查 SOP。
- MemoryGateway 按 tenant/open_id/chat_id/project/requisition/candidate scope filter 检索。
- 公司当前事实必须由 SearchGateway source_refs 支撑。

### Upgrade 7：真实端到端联调

目标：完成服务器、飞书、Bitable、OpenAI/DeepSeek 和 Postgres checkpointer 联调。

验收：

- `docker compose up -d --build` 后 API、worker、postgres、caddy 正常。
- 飞书后台能保存两个回调 URL。
- 测试群 @机器人能收到任务确认卡。
- approve 后 graph 启动，War Room 更新，人工确认后 Bitable 同步。
- 所有未完成外部联调继续标注“未验证”。

## 固定复审清单

- 公开入口只暴露 `/health`、`/feishu/events`、`/feishu/card-actions` 和后续明确允许的 optional adapter。
- 内部控制面必须带 admin key。
- 生产环境不暴露 `/docs`、`/redoc`、`/openapi.json`。
- Agent 不持有飞书、Discord、模型或数据库写 secret。
- API Key 不出现在日志、War Room、AgentRun、profile list 或错误响应。
- 业务副作用全部经过 `interrupt()`、HumanApproval、outbox 和幂等键。
